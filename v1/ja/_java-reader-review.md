# Java 中堅エンジニア読者レビュー（前編 Part 0〜9）

> レビュアー設定：Java で普通に機能開発できる中堅エンジニア。ジェネリクス（`List<String>`）は
> なんとなく使えるが「型消去」は知らない。NullPointerException は日常的に踏むが型システムの言葉
> で説明できない。部分型・変性・健全性・gradual・双方向・単一化などの型理論専門語はすべて初見。
> 継承・インターフェース・`instanceof`・キャスト・オーバーロードという Java の足場は持っている。

### Part 5 ― ナローイング

**生の声**：「ここが一番「なるほど！」と思った章。Java の `if (x != null) { ... }` を書く習慣の
型チェッカー版という説明コラムは、Java 読者に刺さる。`NullPointerException` が型で防げるバグ
だった ─ この言い換えは鮮明。」

「`remove_nil` と `narrow` の実装は、Ruby を知らなくてもロジックが追える。Java 読者にとっては
『条件分岐の枝ごとにスコープを複製して型を差し替える』というパターンが少し独特だが、コードを
読めば分かる。」

「Java・C# の網羅性チェック（missing arm）と Rigor の unreachable arm（到達できない枝の報告）
との比較表が出る。これは Java 読者には直接響く。Java の `switch` でも `instanceof` の
`Pattern Matching`（Java 16+）でも網羅チェックが強制されることがあるので、「逆を向いている」
という説明は腑に落ちる。」

「`possible?` ガードのコラムは実装メモとして脚注扱いで、読み飛ばしやすい。問題なし。」

「**Part 4→5 の接続**について：Part 4 末の予告（『次章ではその逆 ─ Union を減らすナローイング』）
は自然に繋がっており、分割点で話が宙に浮く感じはない。ただし Part 4 では `if` の型付けを
『then 節と else 節の型をまとめる』と書いただけで（ナローイングなし）、Part 5 でその `IfNode`
の実装を「差し替える」形になる。このとき『Part 4 の `IfNode` 実装は暫定版だった』とは明示されて
いないため、「Part 4 と Part 5 でどこが変わったのか」を読者が自分で追う必要がある。」

## 章をまたぐ連続性・付録ポインタの所見

### Part4→5 の分割点

Part 4 末の予告は「ナローイングを Part 5 で作ります」と明示しており、話は繋がっている。
一方で、Part 4 に書いた `IfNode` のコードが Part 5 で差し替えられることが明示されていない。
読者が「Part 4 の実装は完成品」と思って読んでいると、Part 5 で同じノードの処理が出て「あれ、
書き直すの？」となる可能性がある。

### Part 8 の 2 トピック統合

前半（RBS 読み込み）と後半（def 戻り型合成）は、概念的には「型を外から読む」「型を中から作る」
という対称性があり、それを 1 章に収める意図は分かる。冒頭のゴール文で両方を予告しているのは
適切。ただし前半と後半の「橋渡し」段落（8-3 末尾）がやや短く、後半への転換感が少し急。

### 付録 a1 へのポインタ

Part 1 から Part 9 まで繰り返し `付録 a1` へのポインタが出る。内容は充実しており
「必要な項だけ引いて本編に戻れ」と書いてあるが、ポインタ文の表現が毎回「詳しくは付録 a1」と
ほぼ同じ書き方で、読者が「ここは読まないといけないのか？」と迷うことがある。
ポインタごとに「（読み進めるのに必要ではない）」「（`untyped` の詳細だけ見たい人向け）」のような
分量の案内があると、寄り道の必要性が判断しやすい。
