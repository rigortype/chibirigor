## 構成（設計判断を要する／編集が推奨する改善）

### C1. 分割の継ぎ目（前編 Part4→5）：else なし if の「1 | nil」演習が両章で重複

- **引用**：
  - `little/part4-union.md` 演習2（L101-103）：「else の無い `if cond\n 1\nend` の型を `annotate` で確かめると `1 | nil` になる…`union` がこの 2 つをどうまとめるか…」
  - `little/part5-narrowing.md` 演習1（L251）：「else の無い `if cond\n 1\nend` の型を `annotate` で確かめ、なぜ `1 | nil` になるか説明せよ。」
- **問題**：分割でできた隣接 2 章の章末演習に、ほぼ同一の設問（同じコード・同じ `1 | nil`・同じ確認）が連続して載る。Part 4 は union のまとめ方を問い、Part 5 は narrowing 文脈で問う意図差はあるが、**読者には「さっきやった」感**が出て、分割の継ぎ目が雑に見える。なお Part5 の `else` なし→nil の扱いは Part5 本文 5-2（L121-123）が担当で、Part4 では union 演習として、二重に同じ題材を使っている。
- **修正案**：どちらか一方に寄せる。Part4 は「`Integer | Integer` が `Integer` に畳まれる／入れ子をならす」という union の小道具に純化した演習（演習1・3）が既にあるので、`else` なし→`1 | nil` は **narrowing を学ぶ Part5 側に一本化**するのが自然。Part4 演習2 は削るか、union の重複削除・入れ子ならしを問う別設問に差し替える。

### C2. 純化の影響（前編 Part1）：痩せ過ぎではないが、コラム1本に外部ポインタが集中

- **引用**：`little/part1-literals-and-arithmetic.md` の §1-4 コラム「実 Rigor の `rigor type-of`」（L227-242）と直後の畳み込み予告（L243-245）
- **問題**：Part1 自体は `Const`/`type_of`/`check`/`annotate` を一通り作り切っており、章としての痩せは**許容範囲**（むしろ密度は良い）。ただし §1-4 末に「`type-of`＝付録a3／erasure＝後編／定数畳み込み＝Part2／軸A/B＝付録a1」と**前方ポインタが4本連続**し、初学者の最初の章としてはやや「先送りの掲示板」化している。純化で外出しした結果、ポインタの集中点になった。
- **修正案**：刊行ブロッカーではない。`rigor type-of` コラムの erasure 段落（L239-242）は付録a1/a3・後編P3 でも繰り返されており、Part1 では「Java の型消去とは別物・後編で扱う」の一言に圧縮し、二重定義の `to_s` ブロックは残しつつポインタ密度を1〜2本に間引くと、最初の章の呼吸が楽になる。

### C3. 後編 Part4 の HKT コラム内 fuel 言及と、Part7 の「停止性総括」がやや先食い

- **引用**：`seasoned/part4-recursive-types.md` §4-5 とコラム（L149-216）。「停止性の工学の総括は Part 7」と送りつつ、同章コラムで fuel・進捗追跡・三値の表まで具体に踏み込む。
- **問題**：Part4 は「再帰型 μ と余帰納」が主役。Rigor 別解（HKT/fuel）を**発展ノート**に置く設計は妥当だが、コラム2本（symbolize_names／HKT 条件型の三値）で fuel の具体・三値表まで展開してしまい、「総括は Part7」と送った先取りが厚い。Part7 §7-6 が同じ fuel/予算/余帰納を再度表にするので、**同一題材が2章で2度テーブル化**される。継ぎ目の重複というより、坂の途中で同じ景色を2回見せる印象。
- **修正案**：軸を壊さない範囲で、Part4 のコラムは「fuel で打ち切る・詳細と総括は Part7」の最小に寄せ、三値表（L207-211）は Part7 §7-6 に一本化を検討。発展ノート扱いを徹底すれば、読み飛ばす読者にも本筋（μ＋余帰納）が痩せない。

### N2. little/README.md と part0 の sidebar `order` が両方 `1`

- **引用**：`little/README.md` frontmatter `order: 1`、`little/part0-introduction.md` も `order: 1`。
- **問題**：Starlight の sidebar 設定によっては同 order の並びが不定になりうる。seasoned 側は README=2／parts=11–18 と桁が分かれているのに対し、little は README=1／part0=1 が衝突。
- **修正案**：sidebar 生成設定（autogenerate か手動 order か）次第で無害な可能性が高い。手動 order 運用なら README を別レンジ（例：0.5 不可なので、part を 2 起点にずらす等）にするか、ビルド結果の並びを一度確認。表示確認だけの軽微事項。

### N3. 図キャプションの仮置きマーカー `[図: 後で清書]` が残存

### N4. 後編 README の坂の一覧と本文 Part1 §1-6 の地図が二重

- **引用**：`seasoned/README.md` の章立て表（起点列つき）と `seasoned/part1` §1-6「後編の地図」（L152-170）がほぼ同じ俯瞰を提供。
