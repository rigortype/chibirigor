# Ruby/Rails ジュニア読者レビュー（前編 Part 0〜9）

**レビュー想定読者プロファイル**：Ruby で入門し Rails でアプリを作れるジュニアエンジニア。
`String`/`Integer`/`Array`/`Hash`/`Data.define` は実物として理解。`class`/`nil`/`if`/例外/ブロックは日常。
一方で「型を*書く*」「コンパイル時に型が合う/合わない」経験が無く、**「型＝クラス」と同一視している**。
型注釈・静的型・ジェネリクス・部分型・union/narrowing・RBS・健全性は概念ごと初見。TypeScript 構文も読めない。

**重大度基準**：
- **BLOCK** … 詰まって先へ進めない。著者による補足か構成変更が必要。
- **FRICTION** … 引っかかるが文脈・後続で回復可能。注記の追加か言い換えで解消できる。
- **nitpick** … 細かい表現の気になり。必須ではない。

## Part 0 ― はじめに

### 所見

**FRICTION-0a（「型注釈」の導入時点）**
0.1 節の脚注「型注釈＝この変数は `Integer` のように型をコードに書き添えること」は、読者にとって
初めて「型注釈」という行為を聞く場面。脚注形式＋「Part 8 まで書かない」という宥め文句は十分に
機能している。FRICTION 未満に抑えられている好例。ただし「Ruby では普段書きませんよね」という
確認が、書いたことが無い読者には「なんで書かないのかも知らないが、書かないのか」と不思議感が残る。
→ 修正は不要。現状でほぼ読める。

**FRICTION-0b（`untyped` の付録ポインタ）**
1.1 節末尾で `untyped` を初出させた直後に「詳しくは付録 a1」と送っている。付録 a1 は「2 つの軸に
分解する」「TypeScript `any`/`unknown` の違い」「他言語対応表」など、かなりの概念密度。読者が
「なんだろう？」と触りに行くと帰ってこなくなる恐れがある。本編から送る前に a1 の想定読者に
「Part 9 を読み終えてから引く参照カタログ」であることが a1 の冒頭に書いてある（「必要な項だけ引いて
本編に戻ってください」）ので大崩壊はしない。とはいえ Part 0 の読者がここで a1 を開くのは早すぎる。
→ severity: FRICTION。「詳しくは付録 a1（Part 9 後に引く）」など一言添えれば解消。

**nitpick-0c（TypeProf コラム）**
「TypeProf は呼び出し元から引数の型まで逆算する」という説明は、型注釈のメンタルモデルが無い読者に
とって「呼び出し元から型を逆算する」という行為が具体的に想像できない。おまけコラム扱いなので
詰まりにはならない。

**全体評価**：Part 0 は概念の地図として機能している。`untyped`・推論・型注釈の「後で書く」保留が
適切に張られている。BLOCK なし。

## Part 1 ― リテラルと算術

### 所見

**BLOCK-1a（`Const[1]` の意味と「型とクラスの違い」の未解決感）**
1-1 節で `Const = Data.define(:value)` を定義し「`1` という値そのものを型にします（`Const[1]`）」と
説明する。読者には「型＝クラス」の素朴像がある。`Integer` はクラスとして知っている。`Const[1]` は
クラスでもなく、`Integer` のインスタンスでもなく、**「型を表すデータ」**である ― このズレが
ここで初めて生じる。

三題噺の②「Ruby は `1` は `Integer` とまでしか言いません」→③「Rigor は `1` という値そのものを型にします」は
前後の対比が鮮やかで良い。しかし「なぜ `Integer` という既知のクラスではなく、新しいデータ
（`Const[1]`）で表すのか」の答えが「後で `case` の枝分けや定数計算で効いてくる」と先送りに
なっている。後続（Part 4・5）で確かに回収されるので設計としては正しいが、この時点で読者が感じる
「型はクラスとは別のデータだったのか」という驚きへの対処が薄い。

→ severity: FRICTION。「型はクラスそのものではなく、型を表す*データ*です。Ruby の `Integer` クラスは
存在しますが、チェッカーの内部では `Nominal[:Integer]` というラベルで表します。`Const[1]` は
『値 1 というピンポイントの型』を表す別のラベルです。」などの一行が 1-1 節冒頭にあると、「型≠クラス」
の接地点になる。

**FRICTION-1b（`Nominal[:Integer]` の説明の薄さ）**
1-1 で `Const`・`Nominal`・`Dynamic` の 3 つを同時に定義し、コードブロックの括弧内説明に
「名前付きクラスを表す型。例: `Nominal[:Integer]`（1-2 の「丸め」で使う）」と添えているだけ。
`Nominal` が何者か、`Const` との使い分けがここでは不明。1-2 の「丸め」の説明まで行くと分かるが、
1-1 を読んでいる時点で「`Nominal` は何に使うのか」が宙に浮く。
→ FRICTION。1-1 の定義時に「`Nominal[:Integer]` は『Integer クラス全体』を表すラベル（`1`・`2`・
`42` などを全部まとめた大きめの型）」と一言あれば十分。

**nitpick-1c（erasure コラム）**
1-4 の `rigor type-of` コラム内で「Java ジェネリクスの型消去（type erasure）とは別物」と言及。
Java エンジニア向け注意書きとして正確だが、Ruby ジュニア読者には「Java ジェネリクスの型消去」
自体が初耳なので素通りする。おまけコラムなので問題なし。

**好例**：`type_of` が「失敗しない」「わからなければ `Dynamic` を返すだけ」という設計原則を
1-2 の「覚えておく原理」ボックスで早めに固定しているのは、「型チェッカー＝エラーで止まるもの」
という先入観を早期に解除している好例。

## Part 2 ― メソッド送信とディスパッチ

### 所見

**FRICTION-2a（「関数型」という参考書用語との乖離）**
冒頭 blurb で「『しくみ』3 章「関数型」（TAPL 9 章「単純型付きラムダ計算」）に対応します」と書いている。
Ruby ジュニア読者には「関数型プログラミング」との混同を招く可能性がある。この参照はオプションで、
読み飛ばしを促す書き方になっているので FRICTION 未満。ただし「あの本は関数の型を
`{ params, retType }` というデータで持ちました」は、「あの本の読者向けの説明であって、読んでいない
自分には関係ない」と正確に判断できるため問題なし。

**nitpick-2b（dispatch 5 段カスケードの付録ポインタ）**
2-2 節で「本物の Rigor は 5 段カスケードで引きます（詳しくは付録 a3）」と送っている。
おまけ扱いのポインタとして機能しており、本筋を止めない。問題なし。

**好例**：`class_of` が `Const[1]` も `Nominal[:Integer]` も `:Integer` に丸めて表引きするという
操作を、Ruby の `case/when` で実装して見せているのは、読者がすでに知っている `case` の構文を
足場に使えており親しみやすい。

## Part 3 ― ローカル変数と不変 Scope

### 所見

**FRICTION-3a（「型環境 tyenv」という型理論用語の突然登場）**
冒頭で「型環境（Scope）」「`tyEnv`」と書いている。『しくみ』を読んでいない読者には「型環境」が
初耳。ただし直後に「ただの『変数名→型』の対応」と言い換えており、コードでも `@locals = {}`
というハッシュで実装されているので、「`Hash` で変数と型を対応させるだけ」と接地できる。
→ severity: FRICTION（軽度）。「型環境（Scope）―― ただの「変数名 → 型」の対応」という
書き方は既にそうなっているので追加対処不要。読み続けて回復できる。

**FRICTION-3b（「不変」の意味が Rubyist に刺さりにくい）**
3-1 節で「Scope は不変（immutable）である」という設計を強調し、`with_local` が「新しい Scope を
返す」ことを解説している。Ruby ジュニア読者には、`Hash#merge` が新しいハッシュを返すことは知っている。
しかし「なぜ不変でないといけないか」の動機（分岐後に片方の枝の状態が他方に漏れない）が説明されていない。
三題噺の③に「不変設計（本物の Rigor も同じ不変設計）」と書くだけで理由が薄い。
→ severity: FRICTION。後続の Part 5（then_scope/else_scope を別々に作る場面）でこの不変性が
活きる場面になれば自然に腑に落ちる。前に出す必要はないが、3-5 または次章予告に「Part 5 で
`then_scope`・`else_scope` を別々に作るときに不変が効く」を添えれば伏線として機能する。

**好例**：3-2 の「裸の名前はメソッド呼び出し」のメモは、Ruby ジュニア読者が意外に気づかない
挙動で、ここで指摘されると「たしかに！」となる好例。

## Part 4 ― Union

### 所見

**FRICTION-4a（Union 概念の導入は読者フレンドリーだが、「型がクラスではない」のさらなる拡張が無言）**
4-1 節の例 `x = rand < 0.5 ? 1 : "a"` → 「`Integer` か `String` のどちらか」は直感的で分かりやすい。
`Union = Data.define(:members)` もすでに学んだ `Data.define` の応用なので受け入れやすい。

ただし「型＝クラス」の読者には、「`Integer | String`」という記法がすでに Type の 1 つとして
出てくるが、これが TypeScript/RBS の `Integer | String` 記法と同じ見た目になる（`to_s` の出力）。
`Union#to_s = "Integer | String"` という表示は RBS 記法の先取りで、Part 8 まで本物の RBS は
出てこないのに、出力だけ先に目に入る。
→ severity: FRICTION（軽度）。`to_s` で `|` が出ることへの一言「これが後で RBS で見る書き方と
同じ見た目になります」は Part 8 での接続を容易にするが、今は「これが表示形式」と受け取れば十分。
大きな詰まりではない。

**好例**：「`Union` のメンバには `nil` も普通に並びます」「`User | nil` ― 見つかれば値、なければ nil」
という説明は、Ruby ジュニア読者が日常で書く `user = User.find_by(...)` の戻り値イメージと直接
接続できる最高の足場。Part 5 への橋渡しもスムーズ。

**nitpick-4b（ボトム型のポインタ）**
「絞り尽くして候補がゼロになった枝には**ボトム型**（`never`）という名前があります。
本書本体では型としては作らず、付録 a1 に」という注釈は、前に「`never` 聞いたことない」読者には
素通りでOK。問題なし。

## Part 5 ― ナローイング

### 所見

**FRICTION-5a（「ナローイング」という語が突然出る）**
章タイトルと冒頭で「ナローイング（絞り込み）」が登場する。括弧で即「絞り込み」と和訳されており、
`if x.nil?` の例も直後に来る。Ruby ジュニア読者が毎日書く nil チェックの文脈で説明されているので
「型を*絞る*」という直感は掴みやすい。BLOCK ではない。

**FRICTION-5b（「dead branch」の先に「ボトム型」のポインタが重なる）**
5-2 節後半の `is_a?` dead branch の説明→5-3 の 2 つの掟と続く流れで、「dead branch を絞らない」
という判断理由が「誤検知を避ける」としか書かれていない。「なぜ dead branch を絞ると誤検知が出るか」
は `possible?` ガードの実装メモで具体的に示されており（`Integer` 単体に `is_a?(String)` を当てると
`x + 1` が「String の足し算」と誤検知）、これは十分。

→ ここは問題なし。

**好例**：5-1 の「ぬるぽ」コラムは Java 読者向けと明示されているが、Ruby ジュニア読者（Rails でも
`nil` 問題に遭遇している）には「nil チェックしないと落ちる、あれを型で先回りできる」という具体的
なメリットが一番わかりやすく書かれている章の山場。このコラムは Ruby 読者にも非常に効く。

**FRICTION-5c（Rigor vs Java/C# 比較表）**
5-2 節末尾の「missing arm vs unreachable arm」比較表は、Java/C# 経験者向け。Ruby ジュニア読者には
「Java/C# の switch 網羅性」が初耳で素通りする。おまけ扱いだが「Rigor の方向性の説明」として
本文に組み込まれているため、少しノイズになりやすい。
→ severity: FRICTION（軽度）。「Java/C# に触れたことがなければ読み飛ばして構いません」的な
一文があれば十分。現状でも先へ進める。

## Part 6 ― ハッシュと配列の型

### 所見

**FRICTION-6a（「部分型」語の先取り）**
6-3 節で「**キーが多い方が部分型**（少ない方の部分型）」と書かれている。括弧に「なお『部分型』自体は
次の Part 7 で『箱に入るか』として正面から説明します」と断りがある。良い先送り。ただし「逆に見える？」
というフォローが、読者の困惑を先に認めた上で「キーが多い＝より多くの要求に通る」と短く言い換えており、
「今はここだけ掴めば進める」という設計になっている。
→ severity: FRICTION（軽度）。「今はここだけ掴めば進める」が明示されているので BLOCK にはならない。
Part 7 で回収されることへの信頼もある。

**好例**：6-1 の `HashShape` が「options ハッシュだらけの Ruby で一番実用的になる」という前振りは、
Rails のコントローラで `params.permit(:name, :email)` などを書いている読者に刺さる。足場として機能している。

**好例**：6-2 で「未知キーは `nil`（エラーにしない）」が「実際の Ruby が `{foo: 1}[:zzz]` で
`nil` を返すから」と Ruby の挙動に直接根拠を置いているのは読者が検証できる説明で、信頼感がある。

## Part 7 ― 受理判定と「たぶん」

### 所見

**FRICTION-7a（「部分型付け」「部分型」の正式導入）**
7-2 節で「部分型付け」が正式に出てくる。括弧内に「用語は覚えなくて大丈夫。*小さい箱は大きい箱に
入る*、とだけ」と言い換えており、「箱に入るか」という直感を一章かけて丁寧に展開している。
コラム「部分型は継承だけではありません（Java 読者へ）」では「Java 読者へ」と明示されているので
Ruby ジュニア読者は読み飛ばせる。しかし「構造的部分型」「名前的部分型」という語が出てくる以上、
知らない語彙が積み重なるのは FRICTION。
→ severity: FRICTION（軽度）。コラムが「Java 読者へ」と明示されているので回避パスがある。
問題は軽度で、「箱に入るか」の直感だけで先に進める流れになっている。

**FRICTION-7b（`widen` の役割説明が薄い）**
7-2 節で `def widen(t) = t.is_a?(Const) ? Nominal[t.value.class.name.to_sym] : t` が登場する。
「`Const[1]` のような『値そのもの』は、いったんクラスに丸めてから比べる」という説明はある。
ただし「なぜここで丸めるのか」（`Const[1]` と `Nominal[:Integer]` を直接 == 比較すると一致しないから）
という理由が書かれていない。Part 1 の「丸め」の説明と対応しているが、Part 1 で丸めは「足し算の結果を
`Integer` に丸める」文脈で出ており、`accepts` での丸めと文脈が違う。
→ severity: FRICTION。「Part 1 で `Const[1]` を `Integer` に丸めたのと同じ要領で、比較前にクラスに
合わせます」などの一行があれば Part 1 との繋がりが見える。

**好例**：7-4「`:maybe` は罰しない ── この章の山場」での「エラーは期待型が決まっている所でしか出ない。
そして untyped が混ざれば `:maybe`、`:maybe` は罰しない」というまとめは、前編全体の哲学を一文に
凝縮した名文。Ruby ジュニア読者がここまで読んでいれば、「型チェッカーが静かな理由」が腑に落ちる。

## Part 8 ― RBS と型シグネチャ

### 所見

**BLOCK-8a（RBS 記法の初登場と読み方の説明）** ← 最重要
8-1 節で RBS 記法が初登場する：
```rbs
def +: (Integer) -> Integer
```
直後に「読み下しておきます」とあり「`Integer#+` は Integer を 1 つ受け取り、Integer を返す」と
説明している。この説明自体は丁寧。

しかし読者にとって「型注釈を書く」経験がここで初めて来る。Part 0 の約束「Type の注釈を書くのは
Part 8 の RBS まで先で、それまでは書きません」がここで回収される。問題は「RBS は Ruby のコードでは
なく、`.rbs` という別ファイルに書く」という説明の直後に、なぜ別ファイルに書くのかの動機が
十分に示されていない点。「Ruby のコードには型注釈を書きません。型は RBS という別ファイルに書きます」
と言われたとき、読者が感じる「なぜ？」への答えが「Ruby/RBS の世界観です」だけでは薄い。

→ severity: BLOCK（詰まりとして先へ進めなくなるほどではないが、「なぜ別ファイルか」の疑問が
残ったまま進む事になる）。実際には 8-2 で「依存を増やさず何が起きているか全部見える」と
自前実装を選ぶ理由を示しており、RBS の意義自体は流れで掴める。ただし「コードと型を分離することで、
型チェッカーを後付けで足せる（Ruby のコードを変えずに型情報だけ追加できる）」という一文があると、
読者の「なぜ？」が先に消える。
→ BLOCKからFRICTIONへの格下げ候補：本文の流れでほぼ掴めるが、一文の補強で確実に解消できる。
→ **FRICTION として記録する**。

**FRICTION-8b（`def greet: () -> String` の `: ()` 部分の読み方）**
8-1 の読み下しで「`def to_s: () -> String` なら『引数なしで `String` を返す』」と書いているが、
`: ()` の `:` は何か、`()` が「引数なし」を意味することの説明がない。Ruby の `def to_s` との対比
で「`:` の右が型」と説明しているが、「`:` で型シグネチャ区域が始まる」「`()` は空の引数リスト」
という構文の読み方が一行説明だけ。
→ severity: FRICTION。読者は「`:` は何だ？」と一瞬止まるが、`(Integer) -> Integer` と
`() -> String` を比べれば「`()` が引数なし」は文脈で分かる。先に進める。

**FRICTION-8c（「differ 置換」という用語）**
8-3 節末尾で「differ 置換（外から見た挙動を変えずに内部実装だけ入れ替えるリファクタの呼び方）」と
括弧で説明されているが、「differ」という語は通常の用語ではない（おそらく「transparent replacement」
や「behavior-preserving refactoring」のことと思われる）。読者が検索しても出てこない可能性がある。
→ severity: nitpick（定義が括弧に入っているので実害なし）。「ふるまいを変えない差し替え」と
日本語で言い切るか、カッコに説明を入れる現行の書き方を維持するだけで十分。

**好例**：8-4 で「引数は `untyped` なので誤検知しない（`untyped + Integer` は `:maybe` → 黙る）」と
Part 7 の三値 accepts を自然に回収している接続。前の章で作ったものがここで効く経験が積み重なっており、
学習の流れとして機能している。

## Part 9 ― gradual の哲学

### 所見

**FRICTION-9a（「gradual typing」という名前の突然の正式化）**
9-5 節で「漸進的型付け（gradual typing）」「Siek & Taha の gradual typing（2006）」が出てくる。
前編を通じて「わからない所は黙る」「untyped に逃がす」という実践を積んできた読者には、ここで
「これが gradual と呼ばれていたのか」と名前が後付けで付く体験になっており、流れとして自然。
→ severity: FRICTION（軽度）。名前が先に来ないからこそ体験後に名付けられる形になっている。

**好例**：9-2「わざと見逃している 4 箇所」の総括は、前編を通じて「なんとなく黙っている」と感じて
いた箇所に、明示的な理由付けが与えられる節。読者が「型チェッカーなのになんで怒らないの？」という
疑問を持っていた場合、ここで全部回答される。前編のブックエンドとして機能している。

**FRICTION-9b（特別な型 3 種の総括テーブルに「`unknown`」が唐突に出る）**
9-4 の表ヘッダ行では「`untyped`（`Dynamic[Top]`）」が来るが、付録 a1-5 のテーブルでは同じ行が
「`untyped`（`unknown`）」になっている（付録ファイル参照）。前編 Part 9 の本文表では
「`untyped`（`Dynamic[Top]`）」なので問題なし。付録内の揺れは nitpick。

**好例**：9-3 の baseline が「行＋メッセージ」で列を含めない理由説明は、実装の細部を「編集しても
baseline が外れないように」という現場感覚で説明しており、Rails ジュニア読者には「CI で出続ける
古い警告を一旦無視する」経験と接続できる。

## 付録 a1 の精読メモ（参照として）

付録 a1「特別な型カタログ」は「引く」ための整理と明示されている。
a1-1 の「2 軸に分解する」（軸 A：部分型格子の位置／軸 B：チェックするか/しないか）は、
TypeScript `any`/`unknown` の違いを例示しており、TypeScript 未経験の読者には概念の比較対象が
存在しない。ただし「`unknown` は分からないから絞れと迫り、`any`/`Dynamic` は分からないから黙る」
という一文は TypeScript を知らなくても対比として理解できる。**付録を早期に読んだ場合の負荷が高い**
ため、Part 0〜1 からポインタを飛ばす際の一言警告（「Part 9 後に引く参照用」）は重要。
前編 Part 9 の総括まで読んでから a1 に来れば、概念の負荷は格段に下がる。

## 総評

### 全体として読める文書か

**読める。** BLOCK 判定ゼロは、前編の構成として大きな成果。「型≠クラス」の最初のズレ（Part 1 の
`Const[1]`）から始まって、各章が一つの概念だけを足す積み上げ構造は、型注釈経験なしの読者でも
「実装を追いながら型の意味が分かる」体験を提供できている。

### 特に良い箇所

1. **Part 0 の「Part 8 まで型を書かない」保留宣言**：型注釈初見の読者の焦りを早期に解除。
2. **Part 4 の `User | nil` 例**：日常の Rails コードと直接接続できる最良の足場。
3. **Part 5 の「ぬるぽ」コラム**：nil 問題を知っている Ruby 読者に「型で防げる」ことを一番
   具体的に見せている山場。
4. **Part 7 の「`:maybe` は罰しない」まとめ**：前編の哲学が一文に凝縮されている。
5. **`Data.define` を型のキャリアに使う設計全体**：読者がすでに知っているクラスを足場に、型を
   「ただのデータ」として実装する体験が積み重なる。
