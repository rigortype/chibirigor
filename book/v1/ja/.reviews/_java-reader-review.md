# L2 伝・Java中堅読者 再査読（2026-06-11／改修後）

読者像：Java で普通に機能開発できる中堅。`List<String>` はなんとなく使えるが「型消去」は知らない。
NPE は日常的に踏むが型システムの言葉では説明できない。部分型／変性／健全性／gradual／双方向／
単一化は全部初見。継承・インターフェース・`instanceof`・キャスト・オーバーロードという Java の
足場は持っている。

判定軸：「彼の既知（ジェネリクス・NPE・継承・キャスト・null）から橋が架かっているか」。
重大度：**詰**（詰まって先に進めない）／**混**（混乱したが推測で進めた）／**nit**（nitpick）。

---

## Part 0 ― はじめに

助かる。

## Part 1 ― リテラルと算術

| 原文引用 | 何が飛んだか | 重大度 |
|---|---|---|
| 「**型は `Integer` のような「クラス」そのものではなく、型を表す*データ*です**」 | ここは Java 読者に**一番効く橋渡し**でもあり、同時に**一番つまずく**所。Java では型＝クラス（普段は型注釈＝クラス名）。「型はデータ（オブジェクト）」という頭の切り替えを明示してくれたのは助かったが、**なぜクラスでは足りないのか**が `Const[1]`（値そのものを型に）の例まで読まないと腑に落ちない。「`1` という値が型になる」は Java に対応物ゼロ（`int` リテラルが型になることは無い）で、最初は「何のために？」と引っかかった。続く `case` の伏線で推測進行。 | 混 |
| 「`Nominal`（名前的型）」「`Nominal[:Integer]`」 | **「名前的（nominal）」が定義なしで導入**。Java 読者は実は「nominal typing＝名前で型が決まる（Java の継承がそう）」の世界の住人なのだが、その語が初出で、Part 7 で出る「構造的部分型（structural）」との対比が無いので「名前的って何の名前？」で一瞬止まる。`to_s` が `:Integer → "Integer"` を出すコードで「ああクラス名のことか」と推測回収。 | 混 |
| 「`Dynamic`（`untyped`）に逃がします ― 「知らない・確かめようがない ─ 黙っておく」の印」 | コラム「分からないには正反対の2流派」で TypeScript の `unknown`/`any` と対比してくれるのは良い。ただし**Java 読者にとっての対応物（`Object`？ raw type？）への言及が無い**ので、`untyped` が「`Object` 型みたいなもの？」という誤った橋を架けかけた。「`Object` とは違う」の一言が欲しい（付録 a1 行きだが本文だけだと迷子）。 | nit |
| 「結果は `Integer` に丸めます」「**どこかで手放して大ざっぱな型に戻す**必要がある」 | 「丸め（widening）」の動機（`x + 2` だと値が分からないから）は丁寧で良い。橋は架かっている。 | （良）|

良かった橋渡し：「`1 + 2` の `+` もメソッド送信」は Java 読者には新鮮だが、`1.+(2)` の併記で
すぐ飲み込めた。`type_of` が `case node` で分岐するのは Java の `instanceof` 連鎖／switch と同型で
親しみやすい。

## Part 4 ― Union

| 原文引用 | 何が飛んだか | 重大度 |
|---|---|---|
| 「**無タグの合併型**」「持っているのは値に札を付けて区別する*タグ付き*の variant」 | **「タグ付き variant」と「無タグ Union」の対比が定義なしで導入**。Java 読者にとって「Union 型」自体が初見概念（Java に union 型は無い。`sealed interface` + `instanceof` で擬似的にやる程度）。「タグ付き variant」が何を指すのか（Java の sealed class? enum?）見当がつかず、「付録 a5-4 へ」で飛ばされるので**本文だけでは対比の意味が宙に浮く**。幸い Union 自体の説明（「Integer か String のどちらか」）は明快なので、対比部分は読み飛ばして進めた。 | 混 |
| 「`x = rand < 0.5 ? 1 : "a"`」で「`x` の型は…どちらにもなり得る」 | Union の動機づけは明快。Java 読者は「三項で型が違う枝→共通スーパータイプ（`Object` 等）に上がる」のが当たり前なので、「Union として両方覚える」のはむしろ嬉しい驚き。橋は架かった。ただし**「Java だと共通の親まで上がってしまう（情報が失われる）が、Union は失わない」という対比**があれば刺さったはず。 | nit |
| 「**ボトム型**（`never`）」 | 4-1 末尾で予告だけされて付録 a1 へ。Java に対応物が無く（`Void`? 違う）、ここでは「ふーん」で流すしかない。本文の理解には支障なし。 | nit |

良かった橋渡し：`union` 小道具の「入れ子をならす・重複を消す・1個なら包まない」は手続きが
具体的で、`Integer | Integer → Integer` の例で完全に納得。

## Part 5 ― ナローイング

| 原文引用 | 何が飛んだか | 重大度 |
|---|---|---|
| コラム「`nil` で落ちるバグは…**型で表現でき、型で防げるバグ**だった。この見方の転換が「**null 安全（null safety）**」」 | **この本で一番効いた橋渡し（白眉）**。Java 読者の「NPE は実行時にたまたま落ちるもの」という直感を名指しで取り上げ「型で防げる」へ転換させてくれる。`User?`／`User | null` との対応（付録送りだが）も添えてある。NPE という既知から null 安全への橋が明示的に架かった。 | （白眉）|
| 「`if x.is_a?(String)` の then 節は `x` を `String` に絞る」 | `is_a?` は Java の `instanceof` と1対1で即飲み込める。さらに **Java 16+ のパターンマッチング `if (x instanceof String s)` で `s` が String に絞られる**のと完全に同じ挙動。中堅 Java 読者には「あ、あれか」と最高に橋が架かる場面。**だが本文はこの Java 対応を明示していない**（`instanceof` への言及ゼロ）。`nil?` で NPE を名指ししたのに `is_a?` で `instanceof` を名指ししないのは惜しい。明示すれば白眉になれた。 | nit |
| 「その枝は dead branch なので触らない」「dead branch」 | **「dead branch」が英語のまま定義なしで多用**（5-2 で複数回、まとめ表でも）。和訳「到達しない枝」は「到達できない枝の報告」見出しでようやく出るが、初出時は英語のみ。文脈で推測はできる。 | nit |
| 「Java・C# の網羅性検査とは逆に、**動くものには黙り、通らない枝だけを指摘する**」 | Java（`switch`/sealed の網羅性）を名指ししてくれて橋が架かる。良い対比。「網羅性検査＝missing arm を書くまで止める」が Java のどの機能か（`sealed`+`switch`）まで具体名があれば更に良かった。 | （良）|
| 「事実は「変数名」ではなく「そのスコープ位置で確定した事実」に結びついている」 | 5-3 掟2の補足。**「スコープ位置で確定した事実」がやや抽象的**で、再代入で全 facts がリセットされる「なぜ」が一読では掴みにくかった。「`x` に別の値を入れたら前の絞り込みはもう信用できない」という直感は分かるが「スコープ位置」という言い回しが引っかかる。推測で進行。 | 混 |

良かった橋渡し：null 安全コラムが圧倒的。ナローイングの図（then/else で型が分かれる）も明快。

## Part 6 ― ハッシュと配列

| 原文引用 | 何が飛んだか | 重大度 |
|---|---|---|
| 「**キーが*多い*方が部分型**」「`{name:, admin:}` は `{name:}` の部分型」 | **Part 6 最大の引っかかり**。Java の継承では「メンバが多い＝サブクラス（部分型）」で、それはサブクラスがスーパークラスの代わりに使えるから。ハッシュで「キーが多い＝部分型」が同じ理屈（多い方が要求に通る＝代入可能）なのは本文の「`{name:}` が欲しい所に `{name:, admin:}` を渡せる」で**かろうじて**橋が架かる。ただし本文自身が「逆に見える？」と認めている通り、**Java の継承の直感（フィールドが多い＝サブクラス、向きは実は一致）と、ここで言う「キーが多い＝部分型」が*同じ向き*なのか*別物*なのか**が判然とせず混乱。「逆に見える」に引きずられて不安になった。 | 混 |
| 「これが「動くコードを脅かさない」の、**構造的な型**での現れ方」 | **「構造的な型」が Part 6 で初出**（定義は Part 7 のコラムまで来ない）。ここで「構造的」をいきなり使われると Java 読者は「構造体（struct）の型？」と誤読しかける。Part 7 の structural subtyping コラムで初めて回収されるが、順序が逆。 | 混 |
| コラム「`HashShape` は Rigor の発明ではない…Hack の `shape`、PHPStan/Psalm の `array{...}`」 | Java 読者には PHP/Hack の文脈が遠いが「同じ問題に皆ぶつかって open を選んだ」という歴史の重みは伝わる。理解の妨げにはならない。 | nit |

良かった橋渡し：`h[:zzz] => nil`（未知キーはエラーにせず nil）の「実際の Ruby が nil を返すから」
という理由づけは明快。Java の `Map.get` が `null` を返すのと同じ挙動なので即納得。

## Part 7 ― 受理判定と「たぶん」

| 原文引用 | 何が引っかかったか | 重大度 |
|---|---|---|

| 「`Const[1] と Nominal[:Integer] をそのまま比べても一致しません…値そのものの型とクラスの型は別物」「`widen`」 | Part 1 の「丸め」が `widen` という名前でここで再登場。橋は架かっている。 | （良）|
| 「Union のメンバ全員が通って初めて `:yes`（一番弱い結論を採る＝**union-subtyping**）」 | **「union-subtyping」が英語のまま定義なし**で①型理論欄に置かれる。本文の「全メンバ回して一番弱い答え」の説明は明快なので英語タグは無視して進めた。 | nit |
| 「**robustness principle（Postel の法則）**」「返りは厳密・引数は寛容」（7-5＋脚注） | 「Postel の法則」は初見だが具体例（引数は accepts で緩く、戻りは type_of で厳密）が先にあり名前は後付けでも理解できる。脚注「覚えなくて構わない」も親切。橋は架かった。**ただし Java の「オーバーライド時の戻り共変／Liskov 置換」**との対応があれば中堅 Java 読者には一撃だった。 | nit |

## Part 8 ― RBS と型シグネチャ

| 原文引用 | 何が引っかかったか | 重大度 |
|---|---|---|
| 「型は **RBS** という*別ファイル*（`.rbs`）に書きます」「`def +: (Integer) -> Integer`」 | 「型を別ファイルに書く」は Java には新鮮だが「動いてる Ruby を1文字も変えずに型を後付け」の狙いが明快で、記法の読み下し（`:` の右が型、`->` の右が戻り型）も丁寧。橋は架かった。 | （良）|
| 「『しくみ』9 章「**ジェネリクス**」（TAPL 23 章「全称型／System F」）の*型代入*が遠い親戚」 | **冒頭でいきなり「ジェネリクス＝型代入＝System F」と並べられる**。Java 読者は `List<String>` のジェネリクスは使うが、それが「型代入」「全称型」「System F」と同族だとは夢にも思っていない。「遠い親戚」と濁してはいるが、**自分の知ってるジェネリクスとこの章の RBS が何の関係なのか**最後まで本文では繋がらない（実際この章の主題は RBS 読み込みと戻り型合成で、ジェネリクスはほぼ出ない）。見出し語に釣られて身構えたが肩透かし。「この章はジェネリクスそのものは扱わない（後編）」と先に言うべき。 | 混 |
| 「`untyped` の伝播（`n が untyped → n * 2 も untyped`）」「`def double: (untyped) -> untyped`」 | 引数が untyped だと戻りまで untyped が流れる、が `double`/`mystery` の例で明快。「untyped がどこに出るか＝推論の穴」の見せ方は Java 読者にも刺さる。橋は架かった。 | （良）|
| 「**differ 置換**（外から見た挙動を変えずに内部実装だけ入れ替えるリファクタの呼び方）」 | 「differ 置換」は初見だが「リファクタリング（振る舞いを変えない内部変更）」そのもので Java 読者に馴染みがあり、括弧説明もあるので困らない。 | nit |
| 「`void`（「値は返るが当てにするな」）…`-> nil` と違って実装が後で戻り値を変えても**破壊的変更（BC break）にならない**」 | Java 読者には `void`＝「戻り値なし」の馴染み深い語なのに、RBS の `void` は「値は返るが当てにするな」という**Java の `void` とは違う意味**。「Java の void とは違う」と注意してくれないと既知に引きずられて誤読する。実際「値は返るが」で「え、void なのに返るの？」と一瞬混乱。 | 混 |

良かった橋渡し：手書き表→RBS 由来への差し替えで「診断が1つも変わらない＝正しさの証拠」は
リファクタの感覚そのもので納得。戻り型を本体から合成（`"hi".upcase` → String）も明快。

## 総評

### 全体として「橋は架かっているか」
**おおむね架かっている。** 前編の最大の達成は **Part 5 の null 安全コラム**と **Part 7 の構造的

「丸め＝widen」「不変 Scope」「ディスパッチ表」「differ 置換＝リファクタ」も Java の足場から

### 詰まって進めない（詰）箇所
**ゼロ。** どこも推測で先に進めた。前編の「やさしさ」設計は機能している。

### 混乱したが推測で進めた（混）― 改善の余地が大きい順
1. **Part 6「キーが多い方が部分型」と Java 継承の直感のズレ**（最重要）。本文自身が「逆に見える？」
   と認めているのに、Java の継承（サブクラスはフィールドが多い＝向きは実は一致）との関係を
   繋いでいないため「別物なのか同じなのか」で不安になる。「Java の継承でもサブクラスはメンバが
   多い。向きは同じ」と一言接続すれば解消。
2. **Part 8 冒頭「ジェネリクス＝型代入＝System F」の見出し語**。毎日使う `List<String>` を
   「遠い親戚」と持ち出しておきながら章では実質扱わない。期待を煽って肩透かし。「この章は
   ジェネリクスそのものは扱わない（後編）」と先に言うべき。
3. **Part 8 の RBS `void`**。Java の `void`（戻り値なし）と意味が違うのに注意喚起が無い。
   「Java の void とは別」の一言が要る。
4. **Part 9 の「格子／⊤／⊥」**。束論は Java 読者に無い。本文に最小の格子図か、せめて
   「Top＝何でも入る一番上、Bot＝住人ゼロの一番下」の直感だけでも本文に欲しい。
5. **Part 4「タグ付き variant vs 無タグ Union」の対比**が付録送りで宙に浮く。
6. **Part 6「構造的」の初出が Part 7 の定義より早い**（順序逆転）。

`degrade`・`dead branch`・`union-subtyping`・`differ 置換`・`progress + preservation` など、
**初出時に和訳・定義なしで英語タグだけ**置く箇所が散見。文脈で推測でき詰まりはしないが、
「専門用語を定義せず使う」に該当。特に `dead branch` は和訳「到達しない枝」が後から出るので

### 「もう一歩 Java に橋を架けてくれたら」惜しい所
