## 章ごとの通読所感

**Part 1（双方向型付け）**
「前編の `type_of`＝合成 `⇒`、`accepts`＝照合 `⇐`」という地図の与え方は明快で、後編全体の見通しが立つ。Java 読者として `Γ＝Scope` の対応はありがたい。詰まったのは §1-2 の推論規則記法 ― 横線（前提/結論）はまだしも `⊢`（ターンスタイル）が無定義。Java に対応記号が無いので一度立ち止まった。§1-3 の「合成は untyped に倒すから診断が出ない」の論理は、2 つの理由（全域化する合成・寛容な照合）を明示してくれて追える。終盤 §1-6 の後編全体マップは良い。

**Part 2（部分型と変性）**
本書で Java 読者が最も「得をしうる」章。`{name:,age:} <: {name:}`（キーが多い方が部分型）は直感に反するが、「値集合の包含」で納得できた。山場の S-Arrow（戻り共変・引数反変）も論理は追える。ただ §2-4「読み共変・書き反変・不変」が、Java のジェネリクス不変性とワイルドカード（`? extends`/`? super`）という**読者が体で知っている既知**にまったく接続されていないのが惜しい。ここは橋を架ければ最大の good になる。§2-3a の「S-Arrow＝LSP＝robustness principle が同じ規則に収束」は読み物として優秀。

**Part 3（ジェネリクスと型代入）**
`subst`・シャドーイング・変数捕獲・α 同値・erasure。「Java の型消去とは別物」を名指しで切り分けた §3-5 は、Java 読者が知る唯一の type erasure を扱ってくれて大変良い。一方 §3-2 の捕獲バグ例（`<T><U>` 入れ子・`foo<U>`）は密度が高く、ラムダ計算未習だと追いにくい。シャドーイング（易）と捕獲（難）の分量が逆で、難所に足場が少ない。

**Part 4（再帰型：μ と余帰納）**
JSON/木/リストという動機が地に足がついていて入りやすい。§4-4 の余帰納（`seen` で打ち切る）は、グラフ探索の visited set の経験で実装イメージが即湧く、巧い説明。発展ノート §4-5 の HKT（`App[F,A]`）は Java に一階ジェネリクスしか無い読者には対応物が無く記号的に見えるが、発展ノート扱いで TAPL 送りなので設計どおり。

**Part 5（本物の型推論）**
道 A（能力収集／duck）は Ruby の文脈で素直。道 B（型変数＋制約＋単一化）が山場。**「単一化」の語が本文で定義されないまま `unify` コードに入る**のが引っかかった（glossary には有る）。Java の `var`/ダイヤモンド `<>` が内部で単一化をしている、と繋げてくれれば既知の橋になる。型理論コラムの **「ランク」が完全に未定義**なのが最大の詰まり ― ①決定不能性の論証がランクの定義なしには空中分解する。固有名（Kfoury–Wells, Wells）と年号が先に来てランクの中身が来ない。§5-4a の TypeProf 対比（whole-program vs local+catalog）は表が明快で good、ただ「抽象解釈」は語を出すなら一言定義を。

**Part 6（完全な FactStore）**
**後編で最も地に足がついて読めた章。** Java の null チェック後に再代入すると IDE の非 null 推論が消える体験と、stability（事実の寿命）が直結する。6 バケツ・無効化タイミング・join（両枝の共通部分）すべて直感的。唯一 §6-5 の「join と呼ぶが実は格子の meet」という用語注が、格子もjoinもmeetも知らない読者には空回り（脚注に落とせばよい）。

**Part 7（健全性と「わざと unsound」）**
progress/preservation を Java 読者最大の恐怖＝NPE（「型が付いても NoMethodError で落ちる」）に接続したのは秀逸。§7-5 の **`~` 非推移＝untyped がキャストのような無秩序な抜け穴にならない理由** は本書屈指の良い接続（`Object` 経由のキャストの緩さの直感と重なる）。詰まったのは正規化（normalization）― Java に「型が付けば止まる」の足場が無く、`while(true)` で接続してくれれば「実用言語はどれも正規化を手放している」が腑に落ちたはず。停止性の工学（余帰納→fuel→予算の相似形）は明快。

**Part 8（本物の Rigor へ）**
理論ではなく工学の層（プラグイン・キャッシュ・LSP・性能・ADR）への橋。Java 読者にとってはむしろ馴染みのある世界（ビルドキャッシュ・LSP は IntelliJ/Eclipse で体感済み）で、安心して読める着地。ADR への誘導も自然。

## リンク健全性チェック（前編・付録への送り）

後編が前編・付録へ送っている主な参照を、リンク切れ・約束破り観点で確認した：

- 前編 little Part 9 §9-2「わざと見逃す 4 箇所」… **実在する**。後編 Part 7 §7-4 が progress/preservation の言葉で回収する約束は果たされている（前編側も §9-2 で総括済み）。
- 前編 little Part 7 §7-5「返りは厳しめ・引数は緩め」… **実在する**（タイトルどおり）。後編 Part 1 §1-4・Part 2 §2-3a の robustness principle 接続は成立。
- 後編 Part 1 §1-3・Part 7 §7-4 が言う「前編 §1-3 の前振り」… little §1-3 は実在し、テーマ（止まらず・脅かさず・そっと通す）は確かに前振りになっている。**ただし** Part 7 §7-4 は「前編 §1-3 で『型チェッカーはわざと甘く作る』と*前振り*した」と**鉤括弧で逐語のように**書くが、little §1-3 にその文言は無い（実際は「わからない所はそっと通す」「動くコードを脅かさない」）。意味は合っているが、引用形式が誤解を招く軽微な不整合。鉤括弧を外し「『そっと通す』と前振りした」等にすると正確。
- glossary：単一化・型再構築・余帰納・変性・包摂・健全性すべて立項あり。ただし本文（特に Part 5「単一化」）が glossary を当てにして本文内定義を省いている箇所がある ― 読者は本文を線形に読むので、本文側にも一言あると親切。
- 付録 a1（特別な型）・a2（ナローイング）・a4（参考書）・a3（ツール）すべて実在。Top/Bot/`~` の深掘りを a1 へ、個別ナローイングを a2 へ送る判断は妥当（本文の肥大を防いでいる）。

## 総評

後編は宣言どおり formal な巻で、「難しい」こと自体は設計どおり ― そこを文句にはしない。Java 読者の足場（ジェネリクス・継承・NPE・キャスト）から見て、**橋が架かっている所と切れている所の差がはっきりしている**。

**橋が秀逸な所**（このまま残すべき good）：
- Part 3 §3-5 で Java の type erasure を名指しで切り分け
- Part 6 全体（null チェック後の再代入＝stability の直感）
- Part 7 progress/preservation を NPE に接続、§7-5 の `~` 非推移をキャストの無秩序さに接続
- Part 4 §4-4 余帰納を visited set の経験に接続

**橋が切れている／回収されない飛躍**（優先度順）：
1. **Part 5 型理論コラムの「ランク」が無定義**（最も深刻 ― コラムの主張①が空中分解。固有名・年号より先にランクの一文を）。
2. **Part 2 §2-4 がワイルドカード `? extends`/`? super` に触れない**（Java 読者最大の既知の橋を逃している。架ければ最強の good に化ける）。
3. **Part 5 §5-3「単一化」が本文で未定義**（`var`/`<>` への接続で救える）。
4. **Part 1 §1-2 の `⊢` 記号が無定義**（一文で済む）。
5. Part 7 正規化が `while(true)` に接続されていない（実用言語はどれも持たない、が伝わらない）。

いずれも「もっとやさしくしろ」ではなく、**Java 読者が既に持っている具体的な足場（ワイルドカード・`var`/`<>`・`while(true)`・型消去）に一行繋ぐだけ**で橋が架かる類の話。形式の難しさを薄める必要はなく、初出語に最小限の一文（ランク・単一化・`⊢`・ground）を足し、Java の既知概念に明示リンクするのが費用対効果が高い。

軽微：Part 7 §7-4 の「前編 §1-3 で『…』と前振り」の鉤括弧が逐語でないのに逐語のように見える点だけ、引用形式を直すと誠実。
