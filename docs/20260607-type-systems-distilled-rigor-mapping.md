# 『型システムのしくみ』と Rigor 実装の対応レポート

遠藤侑介『型システムのしくみ ― TypeScript で実装しながら学ぶ型とプログラミング言語』
（原題 *Type Systems Distilled with TypeScript*, ラムダノート, 2025）の内容と、Rigor
の実装・設計判断との対応をまとめた読書ノート。

> 本文中に実ファイルの path は含めない。Rigor 側は型カリア名・エンジン構成要素名・
> ADR 番号・仕様概念名で参照する。

---

## 0. 二つの出発点

**本書**は、TAPL（Pierce *Types and Programming Languages*）の中核だけを蒸留し、
TypeScript のサブセット言語に対する**最小限の型検査器**を TypeScript で実装していく
教育書である。最終形でも `typecheck(t: Term, ...): Type` という一本の再帰関数を軸に、
章ごとに `Term` / `Type` の union に case を足していく。立場は **TAPL 流の「静的・健全
（sound）」**で、「未定義動作に陥らないことを実行前に保証する」を旗印にする。

**Rigor** は Ruby 向けの**漸進的（gradual）静的型検査器**で、RBS のスーパーセットとして
実運用を前提に作られている。思想の中心は「動くコードを決して脅かさない
（false-positive discipline）」という、本書とは非対称な価値観である。

本書が最後に「次のフロンティア」として名前だけ挙げて去っていく **gradual typing**
（Siek & Taha [ST06], Siek et al. [SV07]）こそ、Rigor が住んでいる土地である。
本レポートは「本書のゴール地点が Rigor のスタート地点」という関係を軸に据える。

## 1. 第1章「型システムとは」— OK/NG の判定基準は誰が決めるか

本書は「型システムはプログラムの OK/NG を判定するもの」と定義し、その動機を
**未定義動作（undefined behavior, "nasal demons"）の排除＝型安全性**に置く。重要なのは、
本書が「`1 + true` が NG なのは健全性のためではなく、`null` 安全な TypeScript では
『総合的に便利だから』だ」と認めている点で、**判定基準は言語設計者のセンス／
マーケティングが絡むトレードオフ**だと明言する。

**Rigor の対応**：Rigor はこの「トレードオフとしての判定基準」を一段ラディカルに振り
切っている。Rigor にとって最上位の事実は「そのプログラムは動く」であり、最悪ケースの
静的読みで動くコードを赤くするのは設計の失敗（フォルスポジティブ）と見なす。本書が
「便利さで決める」と言った所を、Rigor は「動作実績を脅かさないことで決める」と具体化した
ものと読める。さらに本書が触れた**未定義動作の排除**は、Ruby ではそもそも
`NoMethodError` が決定的に飛ぶため動機が弱く、Rigor の診断は「健全性証明」ではなく
「実用的な早期発見」に寄っている。

## 2. 第2章「真偽値の型と数値の型」— Term/Type の union 表現と typecheck

本書はここで土台を据える：プログラム文字列をパーサ（`tiny-ts-parser`）で**抽象構文木
Term**（`{tag:"add", left, right}` 等の判別 union）へ変換し、`typecheck` が
`switch(t.tag)` で構文ごとに型を返す。型の**内部表現 Type** も
`{tag:"Boolean"} | {tag:"Number"}` というオブジェクト型の union で持つ。注目すべきは、
本書が**実装言語の `boolean` をそのまま型として使わず**、`{tag:"Boolean"}` という
タグ付きオブジェクトで「対象言語の型」を表す点。

**Rigor の対応**：

- **パーサ**：本書の自作パーサに当たるのは Ruby 標準の **Prism**。Rigor は Prism の AST を
  そのまま受ける。
- **Term の union ↔ Rigor の型カリア群**：本書の `type Type = {...} | {...}` に対応するのが
  `Rigor::Type::*` のカリア群。本書が `Boolean`/`Number` の 2 要素から始めたのに対し、
  Rigor は最初から豊富なカリアを持つ — `Constant`（リテラル値型）, `Nominal`（名前付き
  クラスのインスタンス）, `Singleton`（クラスオブジェクトそのもの）, `Tuple`, `HashShape`
  （レコード型）, `Union`, `Intersection`, `Difference`（`T - U`）, `Refined`（述語部分型）,
  `Dynamic`（gradual の要）, `DataClass`/`DataInstance`, `Top`/`Bot` 等。
- **タグ付きオブジェクトで型を表す思想**：本書が `{tag:"Boolean"}` で型を表現したのと同じく、
  Rigor も各型を Ruby オブジェクトのカリアとして表現し、`Combinator` モジュールが
  **正規化を通した構築**を一手に担う。
- **真偽値・数値・リテラル**：ここが本書と Rigor の精度差が最初に出る所。本書の `Number` は
  「数値という一個の型」だが、Rigor は **`Constant[1]` と `Constant[2]` を別の型**として持つ
  （リテラル／シングルトン型）。真偽値も `Constant[true]`/`Constant[false]` が別カリアで、
  `bool` は表示上の正規化（`true | false` の畳み込み）に過ぎない。Ruby の真理性
  （`false`/`nil` のみ偽）は条件分岐エッジ上の**ナローイング**で扱い、本書のように条件式を
  `bool` に固定して処理するのとは方向が異なる。
- **typecheck の構造差**：本書は `typecheck` 一本だが、Rigor は**式の型付け（純粋・非破壊）／
  文の評価（Scope を縫って渡す）／メソッドディスパッチ（多段ティア）**に分離されている。
  本書の `if` 節での「`thn`/`els` の型が一致すること」という判定は、Rigor では「条件式の型を
  返り型の union にする」TypeScript 寄りの方針に近く、`Union` カリアで素直に表現される。

## 3. 第3章「関数型」— 関数型と型環境（tyenv）

本書のハイライトは 2 つ：**関数型 `{tag:"Func", params, retType}`** の導入と、**型環境
`tyEnv: Record<string,Type>`** の `typecheck` への引き回し。`func` 評価では環境を
**コピー**して仮引数を足す（破壊更新は引数のスコープ漏れを生む）。型比較が単純なタグ
一致では足りなくなり、専用の **`typeEq`**（引数名は比較せず、`Func` は引数型と返り型を
再帰比較）が導入される。本章は **TAPL 第9章「単純型付きラムダ計算」**対応。

**Rigor の対応**：

- **関数型の表現**：Rigor には本書のような独立した `Func` カリアは**ない**。メソッド／
  ラムダの契約は **RBS のシグネチャ**として持ち、`RbsTypeTranslator` が RBS 型を
  `Rigor::Type` に翻訳する。呼び出しは `(クラス名, メソッド名)` を RBS 環境やプラグインに
  照会し、オーバーロードを選択・適用する **MethodDispatcher** が担当。ブロックは
  `block_type:` として引き回される。
- **アリティ／引数チェック**：本書の `params.length` 一致＋各引数の型比較に当たるのが、
  Rigor のオーバーロードセレクタ（`required_positionals` 等でフィルタ）と、
  `param_type.accepts(arg_type, mode: :gradual)`。本書の二値の `typeEq` に対し、Rigor は
  **`AcceptsResult`（yes/no/maybe の三値＋理由）**を返す。さらに Rigor はキーワード必須性・
  アリティ・キーワード要件のチェックを持つ（conforms-to 検査をアリティ・キーワード必須性まで
  拡張済み）。本書が「等アリティのみ」と割り切った所を実用言語の都合で精緻化している。
- **型環境 ↔ Scope**：本書の `tyEnv`（不変な `Record`、`{...tyEnv, name:ty}` でコピー）に
  **直接対応するのが Rigor の `Scope`**。Scope は不変スナップショットで、
  `with_local(name, type)` が新 Scope を返す ── 本書の環境コピー規律を、Rigor は
  オブジェクト不変性として制度化している。本書がコラムで繰り返し警告した「環境を破壊
  更新すると仮引数が漏れる」問題を、Rigor は不変設計で構造的に封じている。

## 4. 第4章「逐次実行と変数定義」— 文・スコープ・正規化可能性

本書は `seq`（逐次実行）と `const`（変数定義）を追加。新しい**型**は増えず、環境の引き回し
（`const` は `{...tyEnv, name:ty}` で `rest` を検査）だけが論点。本書はここで深い注意点を
2 つ落とす：(1) TypeScript の前方参照は実行時に TDZ クラッシュし得るので、**TS の型
システムは厳密には型安全でない**こと、(2) このサブセット言語は**正規化可能（accepted
program なら必ず停止）**であること（TAPL 第12章）。本章は **TAPL 第11章 §11.3/§11.5
（派生形式・let 束縛）**対応で、逐次実行も `const` も実は関数から派生できると指摘する。

**Rigor の対応**：

- **文の評価**：本書の `seq`/`const` に当たるのが `StatementEvaluator`。Scope を文間で
  **逐次に縫って渡し** `[Type, Scope]` を返す ── 本書の「`rest` を新環境で検査」の Ruby 版。
- **スコープと再定義**：本書は学習目的でシャドーイング処理を**あえて省略**するが、Rigor は
  ローカル再束縛時に、その束縛にひも付くナローイング事実を破棄するなど、実用上のスコープ・
  無効化を扱う（`FactStore#invalidate_target`）。
- **正規化可能性**：本書のサブセットは `any` を持たないため停止性を享受するが、Rigor は
  停止しない実 Ruby を相手にするため停止性は保証できない。代わりに Rigor は**推論予算
  （inference budget）**で発散を抑える方向で対処する（ただし spec の `budgets:` 表は大半が
  未配線で、実体は数個のハードコード上限＋ADR-10 の gem 別予算 ── ADR-41 で配線が計画中）。
  本書の「型システムが正規化を保証する」美しさは、実用検査器では「予算で打ち切る」工学に
  置き換わる、という好対照。

## 5. 第5章「オブジェクト型」— 構造的型付けの起点

本書は `{tag:"Object", props}` を導入し、**構造的（structural）**に型を比較する（`typeEq` の
Object case は `find` でプロパティ名照合 → 順序非依存）。決定的なのは**「プロパティ集合が
完全一致」を要求**し、余剰プロパティを持つオブジェクトを**拒否する**点 ── これを緩めるのが
第7章の部分型付けだと予告する。**TAPL 第11章 §11.8（レコード）**対応。演習では**タグ付き
union（variant 型）**を扱う。

**Rigor の対応**：Rigor は**構造的と名前的（nominal）のハイブリッド**。

- **名前的（既定）**：クラス／モジュール名は名前的 ── `Nominal[String]` は RBS 宣言を真実の
  源として識別される。本書には「名前付き型」が存在しない（純構造的）のと対照的。
- **構造的（境界で）**：RBS の `interface _Closable` 等の**名前付き構造契約**、推論された
  **オブジェクトシェイプ**、**capability role**（`_Each[T]`, `_ToS` 等の能力ロール）が
  構造的に照合される。
- **HashShape ↔ 本書の Object**：本書の `{tag:"Object", props}` に最も近いのが
  `Type::HashShape`。ただし Rigor の HashShape は**必須キー／オプショナルキー／open・closed
  の余剰キー方針／read-only マーカー**まで持ち、本書が「完全一致のみ／部分型は 7 章」と
  割り切った所を、最初から幅・深さ・開閉まで備えた実用形にしている。本書の「余剰
  プロパティを剥がして渡せ」という制約は、Rigor では open shape の受理として自然に解ける。
- **タグ付き union**：本書が演習に回した variant 型は、Rigor では `Union` ＋ `Constant`
  シンボルタグ ＋ ナローイングの組み合わせとして第一級に扱われる（ADR-47 の `case/in`
  到達可能性解析がこの上に乗る）。

## 6. 第6章「再帰関数」— 返り型注釈と環境の縫い方

本書は `recFunc`（`function` 構文）を追加し、**再帰には返り型の明示注釈を必須**とする
（自己参照で返り型推論が循環するため）。型は `Func` を再利用。判定基準は「本体から推論した
返り型＝宣言返り型」。実装の肝は**どの環境をどの文脈に渡すか**で、本体には「関数自身を
足した環境」、`rest` には「外側＋関数束縛のみ（仮引数を漏らさない）別環境」を渡す。
**TAPL 第11章 §11.11（一般再帰）**対応。コラムでは TAPL が `recFunc` の代わりに
**`fix` を項として導入**する設計と対比する。

**Rigor の対応**：Ruby のメソッド定義は最初から自己参照可能なので、本書のような「再帰の
ための専用構文」は不要。`def` のメソッドはディスカバリ段階（ScopeIndexer）で発見表に
載るため、本体からの自己呼び出しは自然に解決される。本書が「返り型注釈で循環を断つ」と
した問題は、Rigor では RBS シグネチャがあればそれが返り型を与え、無ければ推論が本体を
辿る（ADR-46 のクロスファイル解析等）形になる。本書の「環境の縫い方の緻密さ」が `Scope`
不変設計に吸収される構図は第3章と同じ。

## 7. 第7章「部分型付け」— 変性（variance）が実装で向きを変える

本書のクライマックスの一つ。`typecheck` の `call` case で **`typeEq` を `subtype` に
差し替える**だけで部分型付けが入る、という劇的な構成。`subtype(ty1, ty2)` の定義：

- **オブジェクトの幅・深さ部分型**：ty2 の各プロパティが ty1 に存在すればよい（ty1 は
  余剰可）＝幅、各プロパティ型は `subtype` で再帰＝深さ。
- **関数の返り型は共変（covariant）**、**仮引数は反変（contravariant）** ── 実装上
  `subtype(ty2.params[i], ty1.params[i])` と ty1/ty2 が**入れ替わる**ことで変性が目に見える、
  というのが本章の眼目。
- `number`/`boolean` は不変（タグ完全一致のみ、ただし反射律で自分自身の部分型）。
- 部分型を入れると条件式の型に **join（結び）/ meet（交わり）**が要るため、本書は割り切って
  **`if` 構文を削除**して回避する。**TAPL 第15章「部分型付け」・第16章（部分型付けの
  メタ理論）**対応。哲学コラムで「健全な型システムが良いプログラムを弾くのは誤検知（false
  detection）であり、型研究はこれを減らして発展してきた」と述べる ── これは Rigor の根本
  思想そのもの。

**Rigor の対応**：ここが本書と Rigor が最も深く握手する章。

- **`<:` 部分型関係**：Rigor は部分型 `<:`（値集合包含、反射的・推移的、`bot <: T <: top`）を
  持つ。本書が `typeEq → subtype` に差し替えたのと同様、Rigor の引数受理は `accepts` を通り、
  その内部が部分型・整合性判定を行う。
- **変性**：Rigor は Nominal 型引数で**読み共変・書き反変・両方で不変**、Tuple/HashShape は
  要素ごと共変、と本書の規則を実用カリアに展開している。
- **本書が削った join/meet**：本書が「`if` を削除してまで避けた」結びの計算を、Rigor は
  `Combinator.union` の正規化として日常的に行う（本書が逃げた所を Rigor は正面突破）。
- **誤検知の哲学＝Rigor の背骨**：本書のコラム「健全だが良いプログラムを弾くのは誤検知」と
  いう一文は、Rigor の "never frighten working code" と完全に同じ思想。違いは徹底度で、Rigor は
  これを**最上位の設計価値**に昇格させ、診断の追加ごとに誤検知コストを重く見積もる運用に
  している。
- **Postel の法則（ロバストネス原則）**：本書には無い Rigor 固有の層。**返り型は厳密
  （証明可能な最も狭い型）／引数は寛容（本体が壊れない最も広い型）**という非対称規律で、
  本書の「共変返り・反変引数」を**作法レベル**にまで具体化したものと読める。

## 8. 第8章「再帰型」— μ型と余帰納（coinduction）

本書は最難関の一つ。**μ型 `μX.{foo:X}`**（型変数 `X` と本体）を導入し、`Type` に
`Rec`/`TypeVar` を追加。`type` 宣言はパーサ（`parseRec`）が吸収し、エイリアス名を実 μ 型へ
置換するので **`Term` は増えない**。本書は**同値再帰（equi-recursive）**を採用（畳まれた型と
一段展開した型が「等しい」。TypeScript もこれ。ML 系は多く同型再帰 iso-recursive）。実装の
核は `typeEq` の格上げ：

1. `expandType`/`simplifyType` で μ を一段展開、
2. 束縛型変数名の**α同値**を名前対応 map `typeEqNaive` で扱い、
3. 単純展開だと**停止しない**（図8.2/8.3）ので、**比較中のペア集合 `seen` を持つ余帰納
   （coinduction）**＝「今証明中のペアを再び問われたら等しいと仮定」で打ち切る。

**TAPL 第20章「再帰型」**対応。

**Rigor の対応**：

- **RBS／Rigor の型言語は再帰型定義をネイティブには持たない**。代わりに Rigor は
  **軽量 HKT（ADR-20, `Type::App`）**で再帰的データ構造を扱う。例：`JSON.parse` は
  `App[:"json::value", []]` を返し、`HktReducer` が
  `Value = Literal | Array[Value] | Hash[String, Value]` という再帰 union へ還元する。
- **本書の `seen` 集合 ↔ Rigor の fuel 予算**：本書が**無限ループを余帰納の仮定集合で止めた**の
  に対し、Rigor は HKT 還元を **fuel（既定64）＋進捗追跡**で打ち切る。「再帰の展開をどう
  停止させるか」という同じ問題に、本書は理論的（coinduction で正しく等価判定）、Rigor は
  工学的（予算で安全側に打ち切り）に答えている好対照。正規化自体は決定的・非循環で、祖先
  チェーンの循環は visited 集合で検出する。
- **同値 vs 同型**：本書が TypeScript 同様に同値再帰を選んだのに対し、Rigor は型エイリアスの
  遅延展開や HKT の URI 参照という、本書が NOTE で「実検査器はこうすることが多い（エラー
  メッセージが読みやすい）」と触れた**現実路線**に近い設計。

## 9. 第9章「ジェネリクス」— 型代入と変数捕獲

本書のもう一つの頂点。**型抽象 `<T>...` / 型適用 `f<number>`**（TAPL の type abstraction /
application）を導入し、`Type` に `TypeAbs`/`TypeVar` を追加。核心は**型代入 `subst`** で、本書は
まず**間違った実装**（`TypeAbs` 本体に無条件再帰）を見せ、二つの落とし穴を炙り出す：

1. **シャドーイング**：内側 `<T>` が外側 `T` を隠す場合は代入を止める
   （`typeParams.includes(tyVarName)` なら返す）。
2. **変数捕獲（variable capture）**：置換型の自由変数が内側の束縛変数と衝突する。修正は
   **`freshTypeAbs` で束縛変数を `U@1` 等のフレッシュ名に α 変換してから代入**。

型変数下での型等価は、第8章と同じ**名前対応 map による α同値** `typeEqSub`。`typecheck` は
「参照可能な型変数集合 `tyVars`」を（第3章の `tyEnv` と同様に）引き回し、`typeApp` で
`subst` を実行。**TAPL 第23章「全称型／System F」**対応。**第26章「有界量化」
（`<X extends {a:number}>`）**は名前のみ言及。演習の**型引数推論**は「TAPL にも無い・正解を
知らない」として解答を付けず、これが**型推論という未踏領域への伏線**になる。

**Rigor の対応**：

- **ジェネリクスの表現**：本書の `TypeAbs`/`TypeVar` に当たるのは、`Nominal` が持つ
  `type_args`（凍結配列）。`Array` は空配列（raw）、`Array[Integer]` は適用済み。
- **型代入 ↔ RbsTypeTranslator**：本書の `subst` の正道版が
  `RbsTypeTranslator.translate(..., type_vars:)`。`type_vars[:Elem] = String` を与えて
  `Array[Elem]` を `Nominal[Array, [String]]` へ再帰置換する。ディスパッチ時に受信側の
  `type_args` と宣言型パラメタ名を zip して `type_vars` を構築。
- **本書の「変数捕獲」問題は Rigor で起きるか**：Rigor の型変数置換は RBS 由来で、本書が
  苦闘した「同名内側抽象の捕獲」は、RBS の型引数解決とインスタンス化境界で扱われる。本書ほど
  一般のネスト型抽象を相手にしないため、捕獲問題の露出面は小さいが、思想（α 変換／フレッシュ名で
  衝突回避）は共通。
- **有界量化**：本書が言及だけした `extends` 相当は、Rigor では構造契約（interface / capability
  role）への適合検査がその役割の一部を担う。
- **型引数推論（本書が「正解を知らない」と投げた所）**：Rigor は**呼び出し地点でのジェネリクス
  具体化**やブロック返り型からの型変数束縛を実装しており、本書が未踏として残した領域に実用的な
  近似で踏み込んでいる（生レシーバの未束縛変数は `Dynamic[Top]` に degrade する fail-soft）。

## 10. 「おわりに」が指す次のフロンティア ＝ Rigor の本拠地

本書の結びは、(1) 型システム＝**型付け規則**（狭義）＋**健全性・正規化などの定理**（広義）、
型検査器＝**規則を機械的に決定可能にしたプログラム**、と整理し、(2) **やらなかったこと**を
列挙する：完全な型推論／有界量化（TAPL 26）／健全性・正規化の証明／アルゴリズム的部分型付け
（TAPL 16）／そして **gradual typing**（[ST06], [SV07]）。

この**「次に読むべき gradual typing 論文」が Rigor の設計の土台**である。Rigor はまさに本書の
終着点の先：

| 観点 | 本書（静的・System F 風） | Rigor（漸進的・実用） |
|---|---|---|
| 判定 | OK/NG の二値 `typeEq`/`subtype` | **三値 yes/no/maybe**（`AcceptsResult`, `Trinary`） |
| 未知の型 | `any` は対象言語に無い（だから正規化が成立） | **`Dynamic[T]`** が静的ファセット `T` を保ちつつ境界を越える。`Dynamic[Top]` が `untyped` |
| 関係 | 部分型 `subtype` 一本 | 部分型 `<:` ＋ **gradual consistency**（`untyped` が絡むと対称・非推移） |
| 状態 | `tyEnv` を再帰でコピー | **不変 Scope ＋ FactStore** をフロー感応に縫う（ナローイング＝型代入でなく事実集合） |
| 価値観 | 健全性（未定義動作の排除） | **誤検知を出さないこと**（動くコードを脅かさない）を最上位に |
| 再帰の停止 | 余帰納の `seen` 集合 | HKT 還元の **fuel 予算**で安全側打ち切り |
| 構造/名前 | 純構造的（オブジェクト型） | **構造的＋名前的ハイブリッド**（RBS 名前的＋interface/shape 構造的） |
| RBS 往復 | （なし） | 型格子は **RBS へロスレス（保守的に）erasure** 可能 |

---

## 11. 総括

- **本書の各章は Rigor の構成要素にきれいに射影できる**：Term/Type union → `Rigor::Type::*`
  カリア、`tyEnv` → `Scope`、`typeEq`/`subtype` → `accepts`/`<:`、`Object` → `HashShape`＋
  構造契約、μ型/`seen` → HKT/`App`＋fuel、`TypeAbs`/`subst` → `Nominal#type_args`/
  `RbsTypeTranslator`。
- **最大の質的差は三点**に集約される：(1) **gradual** ── 本書が伏線として残した `Dynamic[T]`・
  整合性・三値判定が Rigor の中核、(2) **誤検知忌避** ── 本書のコラムが述べた「良いプログラムを
  弾くのは誤検知」を Rigor は最上位の設計価値に昇格、(3) **工学的近似** ── 本書が理論的に解いた
  停止性・正規化・型推論を、Rigor は予算・fail-soft・ベースライン等の実用機構で代替。
- **教育的補助線として本書は理想的**。Rigor の `Scope` 不変規律は本書の「環境コピー」、Rigor の
  変性規則は本書の「共変返り・反変引数」、Rigor の HKT 還元は本書の「μ型の展開と余帰納」と、
  それぞれが本書の最小実装の**実運用版**になっており、Rigor の設計判断（なぜ三値か、なぜ
  `Dynamic[T]` か、なぜ予算で打ち切るか）は、本書の終章が指し示した gradual typing への一歩
  として読み解ける。

## 12. 補遺：`chibirigor` チュートリアルの副読本として使えるか

[chibivue](https://github.com/chibivue-land/chibivue) をモデルに、最小限の Rigor 実装を
ステップバイステップで作る `chibirigor`（簡易型推論器）の副読本として本書が使えるか、の評価。

**結論：理論の補助線（概念辞典）としては最良。ただし章立ての背骨にはしない。**

### なぜ「背骨」にできないか — chibivue 型と本書型の根本差

- **chibivue の教育モデル**＝「*本物のアーキテクチャ*の最小版を、忠実に、段階的に作る」。
  → chibirigor の背骨は **Rigor の実アーキ**（Prism → Scope → ExpressionTyper →
  MethodDispatcher → narrowing/FactStore → `Dynamic[T]`）であるべき。
- **本書の教育モデル**＝「*おもちゃ言語*（TS サブセット）の検査器を作って*型理論を体感*する」。
  アーキは実ツールに似せず `typecheck` 一本。学ぶのは*型理論の概念*で、実ツールの工学ではない。

この差から、構成を章単位で真似ると 3 つの致命的なズレが出る：

1. **本書は「検査器（OK/NG）」、chibirigor は「型推論器」**。しかも本書は型推論を意図的に
   避けており、第9章演習の型引数推論は「正解を知らない」と解答を放棄している。chibirigor の
   中心テーマ（推論）が、本書の最大の空白と一致する。
2. **本書は健全・静的、Rigor は gradual・誤検知忌避で価値観が逆**。本書の「完全一致 → 余剰を
   拒否」「健全性のために弾く」を素直に取り込むと、Rigor の open shape / `Dynamic[T]` /
   "never frighten working code" と衝突する。
3. **本書は単一 `typecheck`、Rigor の肝はパイプライン分離**（式の純粋型付け／文の Scope 縫い／
   多段ディスパッチ）。chibirigor で一番伝えたいアーキを本書は持っていない。

### 章ごとの副読本価値（chibirigor 視点）

| 本書の章 | 価値 | 使い方／注意 |
|---|---|---|
| 1 型システムとは | ★★☆ | 動機づけ。ただし「健全性」基準は Rigor と逆 → FP discipline と*対比*で引く |
| 2 Term/Type union・typecheck | ★★★ | AST→型表現→検査の骨格。Rigor はリテラル型・Prism なので差分を明示 |
| 3 関数型・型環境 tyenv | ★★★ | **型環境＝Scope の原型**。不変コピー規律はそのまま生きる |
| 4 逐次実行・変数定義 | ★★★ | StatementEvaluator（文を縫う）の原型 |
| 5 オブジェクト型 | ★★☆ | 構造的型／HashShape の原型。*完全一致 vs open shape は逆向き* |
| 6 再帰関数 | ★☆☆ | Ruby は自己参照が自然 → ほぼ不要 |
| 7 部分型付け・変性 | ★★★ | `<:`・共変/反変はそのまま転用可。join/meet は Rigor では正規化で実装と補足 |
| 8 再帰型（μ型・余帰納） | ★★☆ | 理論的背景として。Rigor は HKT/fuel で別実装 |
| 9 ジェネリクス・型代入 | ★★☆ | substitution・α同値・変数捕獲は普遍。ただし*推論は扱わない*＝最大ギャップ |
| おわりに（gradual への伏線） | ★★★ | **ここからが Rigor 本体**。最良の橋渡し |

### 推奨する使い方

1. **背骨は Rigor の実アーキ**（chibivue 流の忠実な最小版）。本書は各 step に貼る
   「この step の理論的背景は本書 X 章」という*参照副読本*に徹する。
2. **本書が埋められない 2 つの穴を別途補う**：**型推論**（双方向型付け / 制約ベース /
   Hindley–Milner の初歩 — chibirigor の中心なのに本書に無い）と **gradual typing**
   （[ST06]/[SV07]、`Dynamic[T]`・consistency・三値判定 — 本書はおわりにで指さすだけ）。
3. **逆向きの箇所（健全 vs 誤検知忌避、完全一致 vs open shape、単一型 vs リテラル型）は
   「対比教材」として積極利用**。「TAPL ならこう弾くが Rigor はこう許す、なぜか」は
   chibirigor の哲学を伝える最高の導入になる。
