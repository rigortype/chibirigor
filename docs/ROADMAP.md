# chibirigor ロードマップ

chibirigor 本体（`lib/`・`exe/`）の機能追加方針。**最小・依存ゼロ・gradual・誤検知ゼロ**という
身上を崩さない範囲で、前編/後編の本文と地続きに育てる。各機能は原則「**1 機能 ＝ 1 Part 相当**
（新ファイル 1 つ＋テスト＋必要なら `examples/` スケッチ）」に収める。

## 大方針（複雑さの予算を守る）

- **複雑さは予算制**。1 機能で持ち込む難所は 1 つ。配管が増えるだけで価値が出ない変更はしない。
- **依存ゼロ**（Prism のみ）。テストフレームワークも使わない。
- **gradual・誤検知ゼロ**を最優先。新しい推論・照合を足しても「期待型が `untyped` なら照合しない」
  「分からなければ `untyped` に倒す」を必ず守る（動くコードを脅かさない）。
- **本文と同期**。本体に機能を足したら、対応する前編/後編の章・`examples/`・用語集
  （`draft/glossary.md`）も更新し、`draft/seasoned/ja/examples/check_docs.rb`（ドリフトチェック）を
  通す。用語・表記はサイト（`rigor.typedduck.fail`）準拠（`AGENTS.md` の用語集）。
- **後編スケッチの昇格を優先**。後編で動かした `examples/*.rb`（`unification.rb`・
  `fact_invalidation.rb` ほか）を本実装に昇格させる形を好む。前編/後編の非対称（後編は「読み解く」
  巻）を埋める副作用がある。

## フェーズと優先度

| 機能 | 労力 | フェーズ | 系統 |
|---|---|---|---|
| 定数畳み込み（`1 + 2 → Const[3]`） | S | 1 | 前編を厚く |
| ソース位置＋キャレット表示 | S | 1 | DX |
| 極小プラグインフック | S | 1 | Rigor 忠実 |
| 戻り型照合（RBS 宣言に本体を `⇐`） | M | 2 | 双方向を一段上げる |
| ジェネリクス／要素型（`unification.rb` 昇格） | L | 3 | 本丸（bounded） |

フェーズ 1（S×3）から着手。フェーズ 2 で初めて「双方向が*仕事*をする」。フェーズ 3 は野心的な
ので**範囲を絞って**（下記）入れる。各機能は独立に出荷可能。

## フェーズ 1 ― 小粒で手触りを濃く

### 1. 定数畳み込み（literal folding）― S ✅ 実装済み

- **狙い**：`1 + 2` を `Const[3]` に*畳む*（実 Rigor の `Constant<3>` リテラル精度の縮図）。
- **現状**：前編 Part 1 は `1 + 2` を `Nominal[:Integer]` に*丸めて*いる（「いつ丸めるか＝正規化・
  予算」を教える題材）。本機能はそれを「**畳めれば畳む・無理なら丸める**」に一段精密化する。
- **やる**：両オペランドが*既知値の `Const`* の算術（`Integer` の `+ - *`、`String` の `+ *`）だけを
  畳む。`Const[3]` の表示は `3`。
- **やらない（予算を守る）**：変数をまたぐ評価・ループ内畳み込み・記号計算はしない。**広げ規則
  （widening）**を必ず置く ― 畳み数や文脈（メソッド本体・繰り返し）の閾値を超えたら `Nominal` に
  丸める（`Const` 爆発を防ぐ＝実 Rigor の正規化予算の最小版）。
- **触る所**：`lib/chibirigor/dispatch.rb`（戻り値を引数の `Const` 値から計算する小ステップ）／
  `type.rb`（`Const` の数値演算ヘルパ）。
- **FP**：精度加算のみ（`Const[3]` も `Integer` 引数に通る）。新しい診断は出ない。
- **完了条件**：`annotate("1 + 2") → 3`、`annotate("1 + x") → Integer`（丸め）、widening が効く例の
  テスト。前編 Part 1 の「丸め」節を「畳めれば畳む」に拡張。
- **本文**：前編 Part 1（小節を 1 つ拡張）。

### 2. ソース位置＋キャレット表示 ― S ✅ 実装済み

- **狙い**：診断を `行: メッセージ` から `file:line:col` ＋ 該当箇所を指す**キャレット**へ。DX 効果大。
- **やる**：Prism ノードの `location`（`start_line`・`start_column`・`slice`）を診断に載せ、CLI で
  該当行＋`^^^` を描く。診断ハッシュに `:column`（必要なら `:length`）を追加。
- **やらない**：複数行スパン・色付けは任意（最小はモノクロ）。
- **触る所**：`type_of.rb` の `diagnostic` ヘルパ／`checker.rb`／`exe/chibirigor`（描画）。
- **FP**：表示のみ・ゼロ。
- **注意**：診断ハッシュ形 `{line:, column:, message:}` は baseline 照合（`baseline.include?(d)`）に
  使われる。キー追加は後方互換だが、baseline 比較に `column` を含めるか要設計（位置がズレると
  baseline が外れる → 既存テストは `[:line]`/`.size` のみ参照なので影響なし、を確認）。
- **完了条件**：`1 + "x"` の診断に列番号とキャレットが出る。`exe/chibirigor` の出力例を更新。
- **本文**：前編に「診断を読みやすく」の小節 or DX 付録。実 Rigor の diagnostic-shape／ADR-51 と地続き。

### 3. 極小プラグインフック ― S ✅ 実装済み

- **狙い**：コア（`Dispatch::METHODS`）を編集せず、外から型情報を**足せる**一点拡張。実 Rigor
  ADR-2（拡張 API）の体感デモ。
- **やる**：登録口を 1 つだけ ― `Chibirigor.register_method(klass, name, params:, returns:)` が
  `Dispatch::METHODS` を*非破壊で*上書き合成する。
- **やらない**：プラグインのライフサイクル・DSL・マクロ展開・複数フック種別は持たない（拡張面は
  1 つに限定）。
- **触る所**：`lib/chibirigor/plugin.rb`（新規・小）／`dispatch.rb`（登録分をマージ）。
- **FP**：加算のみ・ゼロ。
- **完了条件**：登録した `(クラス, メソッド)` がディスパッチで効く例のテスト。
- **本文**：後編 Part 8（本物の Rigor へ）に「極小プラグイン」worked example。

## フェーズ 2 ― 双方向を一段上げる

### 4. 戻り型照合（return-type checking）― M ✅ 実装済み

- **狙い**：メソッドに**宣言された戻り型**があるとき、本体の合成型をそれに `⇐`（照合）する。
  「型が宣言と違う戻り値を返す」を検出 ― **`⇐` が*実際に診断を出す*最初の例**で、後編 Part 1
  の双方向理論が初めて手で動く。
- **現状**：`type_of_def` は戻り型を*合成*はする（前編 Part 8 の annotate）が、宣言と*照合*しない。
- **やる**：`check(source, rbs:)` モードを足し、RBS（Part 7 のミニリーダーが読む `class C; def m: (...)
  -> R end`）に載った `def m` の本体最終式を、宣言戻り型 `R` に `accepts(R, body_type)` で照合。
  `:no` のときだけ診断。新エントリ `check_against(node, expected, scope, diag)`（= 既定は
  `accepts(expected, type_of(node))`、subsumption）を導入。
- **やらない**：完全な双方向（全ノードのモード分け）は不要。`⇐` が*仕事をする*のは「戻り型」と
  「（既存の）呼び出し引数」だけに留める。
- **FP**：宣言がある所だけ照合 → **opt-in**（注釈なし＝照合なし＝FP なし）。宣言が `untyped` なら
  照合しない（gradual）。
- **触る所**：新規 `check` 経路（`type_of.rb` か新 `checking.rb`）／`type_of_def`／`rbs.rb`（ユーザ
  メソッドの戻り型供給）。
- **完了条件**：RBS `def greet: () -> Integer` に対し `def greet; "hi"; end` が診断、`def greet; 1; end`
  は通る、宣言なし `def` は無診断、のテスト。
- **本文**：後編 Part 1 に「`⇐` が診断を出す具体例」を追記、または前編に「戻り型を照合する」新章。

## フェーズ 3 ― 本丸（範囲を絞って）

### 5. ジェネリクス／要素型 ― L（bounded）

- **狙い**：後編 Part 3 の `examples/unification.rb` を**本実装に昇格**し、型変数を扱う。最終的に
  `arr.map { |x| ... }` の `x` を要素型に。これが入ると chibirigor は「双方向＋多相の最小版」になり、
  前編の generics と後編 Part 3/6 が lib で一本につながる（本のクライマックス）。
- **複雑さの線引き（重要）**：L なので**段階を割って、各段で出荷可能に**する。予算を脅かしたら
  止めて「後編スケッチのまま」に戻してよい。
  - **5a（M）要素型の*読み*だけ**：`Array[Elem]`/`Hash[K,V]` をディスパッチ表に型変数で持ち、
    `arr.first`/`arr[0]` が `Elem` を返す。ブロックは触らない。
  - **5b（M〜L）ブロック仮引数への押し下げ**：`map`/`each` の要素型をブロック仮引数に `⇐` で渡す。
    ここで `unification.rb` の単一化を本格使用。
  - **5c（任意）戻り型の多相伝播**：`map` の戻りを `Array[ブロックの戻り]` に。
- **FP**：推論が強くなるほど誤検知リスク増 → **gradual ゲートを厳格に**（生レシーバ／未束縛の型
  変数は `Dynamic[Top]`＝`untyped` に degrade。前編の「埋まらねば untyped」を死守）。
- **触る所**：`type.rb`（`Nominal` の `type_args`・型変数）／`dispatch.rb`（型変数の zip・具体化）／
  新規 `unify.rb`（`examples/unification.rb` の昇格）／ブロックの型付け（5b）。
- **完了条件**：5a で `[1,2].first → Integer`、5b で `[1,2].map { |x| x + 1 }` の `x` が `Integer` 扱い、
  生 `Dynamic` 受信は `untyped` に倒れる（FP なし）、のテスト。
- **本文**：後編 Part 3/6 の sketch を「lib に昇格済み」に更新。前編に新章（generics）を足すかは
  5a/5b の手応えを見て判断。

## 非目標（当面入れない＝予算保護）

- 完全な HM（大域単一化・let 多相の自動一般化）。`unify` は局所・自明な範囲のみ。
- 本物のプラグイン機構（DSL・マクロ展開・ライフサイクル）。フックは 1 点のみ。
- LSP・キャッシュ／インクリメンタル・性能最適化・並行解析。
- 再帰型（`Rec`＋余帰納）・FactStore のコア化は**当面 `examples/` スケッチのまま**（demand 次第で
  昇格を再検討）。本物の `rbs` gem 採用（native ext 非互換が壁）も保留。

> これらは「やらない」ではなく「**いま予算を割かない**」。本体が双方向（フェーズ 2）まで育ち、
> 需要が見えたら個別に再評価する。

## 各機能の出荷チェックリスト（共通）

1. `lib/` 追加は最小（1 機能 1 難所）。新規ファイルは命名・分割を既存に合わせる。
2. `test/test_part*.rb` 相当の依存ゼロ・文字列ソーステストを足す（全 part 緑を維持）。
3. FP ゼロを確認（「動くコードを脅かさない」例を最低 1 つテスト化）。
4. 対応する本文（前編/後編）・`examples/`・`draft/glossary.md` を更新。**機能の反映は「発展ノート
   方式」**（章の教えるコードはその場で書き換えず、章末近くに prose の発展ノートで重ねる。元コードと
   *実物の道具*の挙動がズレるなら橋渡しの一文を添える）。先例＝前編 Part 1 の「1-4b」「1-4c」。
   詳細は `AGENTS.md`「本文への機能反映（発展ノート方式）」。
5. `ruby draft/seasoned/ja/examples/check_docs.rb` が緑（本文コードと実装の同期）。
6. 用語・表記はサイト準拠（`AGENTS.md` の用語集。キャリア／型チェッカー／ナローイング 等）。
