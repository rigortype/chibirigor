# `chibirigor` チュートリアル ― 章立てドラフト

[chibivue](https://github.com/chibivue-land/chibivue) をモデルにした、最小限の Rigor を
ステップバイステップで実装するオンラインブック／チュートリアルの設計ドラフト。

- **ねらい（サブテーマ）**：**型理論のトピック ↔ Ruby/RBS の型システム ↔ Rigor 実装が
  ぶつかった問題**、この三つを*自然に結びつける導入*にする。各 step は小さな三題噺
  （理論 → Ruby/RBS での見え方 → Rigor が実装で困った所）として読める。
- **トーン**：**とても易しく**。数式・専門用語は本文から追い出し、必要ならコラム or
  「実はこれ◯◯と呼ばれます」の後出しにする。**実装も複雑にしすぎない**（1 step ＝ 難しいこと
  1 つ。複数の難所を同時に持ち込まない。詳細は深掘り考察 v3）。
- **モデル**：chibivue 流＝「*本物のアーキテクチャ*の最小版を、忠実に、段階的に作る」。
  背骨は Rigor の実アーキ（Prism → Scope → ExpressionTyper → MethodDispatcher →
  narrowing/FactStore → `Dynamic[T]`）。ただし*最小版*に徹し、易しさを優先する。
- **参考書**：遠藤侑介『型システムのしくみ』（本文では『しくみ』と略記）を*各 step の理論
  サイドバー*として参照する（対応表は `20260607-type-systems-distilled-rigor-mapping.md` § 12）。
  『しくみ』を背骨にはしない（理由は同ノート参照：『しくみ』は「おもちゃ言語の検査器」、
  chibirigor は「Ruby の型推論器」で教育モデルが違う）。
- **実装言語**：Ruby。パーサは Prism。
- **提供機能**：`chibirigor check`（型診断）と、容易なら `chibirigor annotate`
  （推論結果の出力）。

## Part 0. イントロダクション（v1 アウトライン）

### 0.1 「検査器」と「推論器」は何が違うのか（最初に必ず触れる）

- **型検査器（type checker）**：プログラムに付いた*型注釈*を前提に、それらが矛盾しないかを
  判定する（OK/NG）。『しくみ』が作るのはこちら。対象言語は仮引数に必ず
  型注釈が付く（`(x: number) => x`）TS サブセットで、検査器は注釈を*検査*するだけ。『しくみ』が
  型推論を扱わないのはこのため（第9章演習で「型引数推論は正解を知らない」と明言）。
- **型推論器（type inferencer）**：注釈が無い（少ない）コードから型を*導出*する。Rigor /
  chibirigor はこちらが主役。コードそのものから型を推論し、その推論結果を使って矛盾
  （`check`）と推論型の提示（`annotate`）を行う。
  - **誤解を最初に解く**：ここでいう「推論」は Hindley–Milner（大域制約を単一化で解く）では
    *ない*。Rigor の推論は **構文主導の上向き合成（リテラル畳み込み）＋ フロー・ナローイング
    ＋ シェイプ推論 ＋ シグネチャ参照（RBS）** の合成。メソッドの引数・戻り型はコードから
    solve せず RBS から*引く*。『しくみ』との本当の差は「HM をやるか」ではなく「注釈の無い箇所を
    合成で埋めるか、注釈必須にするか」（→ 深掘り考察1）。
- **本チュートリアルの立場（双方向型付け）**：型付けを 2 方向に分ける ―
  **合成 `⇒`**（式から型を上向きに得る。注釈不要、失敗しても `Dynamic` を返すだけで*診断に
  しない*）と **照合 `⇐`**（期待型に対して下向きに検査する）。`check` ＝「合成した型を `⇐`
  位置で期待型に突き合わせる行為」。
  - **重要な原理**：**診断は照合 `⇐` の位置でしか出ない。** そして `⇐` 位置＝*期待型が
    宣言されている所*（RBS シグネチャ・明示注釈・宣言付き戻り値）に限られる。ゆえに
    **「注釈の無い動くコードは、構造的に、決して脅かされない」** ― これが Rigor の
    "never frighten working code" の形式的裏付け（→ 深掘り考察2・3）。この方向分けは
    robustness principle（返り厳密＝`⇒`／引数寛容＝`⇐`）とも一致する。

### 0.2 Rigor（と chibirigor）の入力スタンス：拒まない

- Rigor は **Ruby が syntax error にしないコードであれば、すでに受け入れる**。型が付かない
  から解析を拒む、ということはしない。
- 実際には Prism を使うため、*解釈可能なコードの範囲は Ruby 実行系よりさらに広い*
  （Prism は壊れた構文も部分的に AST 化する）。ただし **chibirigor は「型が付く＝動作する」を
  保証しない**。あくまで推論と診断を返すだけ。
- これは『しくみ』と決定的に違う：『しくみ』の検査器は NG で*例外を投げて止まる*し、対象言語は厳密な
  サブセット。chibirigor は**入力を常に受理し、出力は診断＋推論型**。この「拒まない／
  脅かさない」姿勢が gradual・FP discipline の出発点であることをここで宣言する。
- サイドバー：『しくみ』 第1章（OK/NG の判定基準はトレードオフ）と*対比*。『しくみ』は健全性、Rigor は
  誤検知忌避。

### 0.3 chibirigor のゴールと提供機能

- `chibirigor check FILE...` … 型診断を出す（Rigor の `check` のミニ版）。
- `chibirigor annotate FILE` …（容易なら）推論した型を出力する。最小実装は「各メソッドの
  推論シグネチャと各ローカルの推論型をインライン or サイドカーで提示」。Rigor の `annotate`/
  dump 系のミニ版。
- *非対象*（最初から線を引く）：プラグイン機構、キャッシュ、LSP、インクリメンタル、
  完全な RBS、健全性証明、性能最適化。

### 0.4 セットアップ

Prism のインストール、最小 CLI スケルトン、`parse` して AST を覗く（『しくみ』 2.2 に相当）。

## Part 1. 最小の型推論器 ― リテラルと算術

**狙い**：AST → 型表現 → 推論 → 診断 の一巡を最小で通す。『しくみ』 第2章の Ruby 版。

- Prism AST の受け取り（`IntegerNode`, `FloatNode`, `TrueNode`, `StringNode`, `CallNode` の `+` 等）。
- 型表現の導入：`Top` / `Bot` / `Dynamic` / `Constant`（リテラル値型）/ `Nominal`（クラス）。
  - 『しくみ』の `{tag:"Number"}` に対し chibirigor は **`Constant[1]`（リテラル型）**から始める
    ことで Rigor の精度思想を最初に見せる。
- `type_of(node) -> Type`（純粋・非破壊。Rigor の ExpressionTyper のミニ版）。
- `check` 骨格：`1 + "x"` のような矛盾を診断にする（例外で止めず*診断を貯める*）。
- `annotate` 骨格：トップレベル式の推論型を出力（『しくみ』の `console.log(typecheck(...))` に相当）。
- 落とし穴：『しくみ』のように NG で throw しない。診断は集めて続行（FP discipline の最初の一歩）。
- サイドバー：**『しくみ』 第2章**（Term/Type union と typecheck の骨格）。

## Part 2. メソッド呼び出しとディスパッチ

**狙い**：Ruby は「関数」ではなく「メソッド」。レシーバ＋メソッド名から戻り値型を引く
ディスパッチを導入。『しくみ』 第3章（関数型）の Ruby 的読み替え。

- `(レシーバ型, メソッド名) -> 戻り値型` のミニ・メソッドテーブル（手書きの組み込み数件）。
- MethodDispatcher のミニ版。アリティ不一致を診断に。
- **gradual の核**：未知メソッド／未知レシーバは*診断にせず* `Dynamic` に degrade する
  （= 動くかもしれないコードを脅かさない）。『しくみ』の「未定義変数は即エラー」と対比。
- サイドバー：**『しくみ』 第3章**（関数型と判定基準）＋「Rigor は関数型カリアを持たず、
  シグネチャで持つ」差分。

## Part 3. ローカル変数と不変 Scope、文の評価

**狙い**：型環境＝Scope。『しくみ』 第3・4章の Ruby 版。

- 不変 `Scope`（`locals`）と `with_local(name, type)`（新 Scope を返す）。『しくみ』の
  `{...tyEnv, name:ty}` コピー規律をオブジェクト不変性として実装。
- `StatementEvaluator`：文を縫って `[Type, Scope]` を返す。逐次・再代入。
- `annotate`：各ローカルの推論型を提示。
- サイドバー：**『しくみ』 第3・4章**（tyenv、環境コピー、逐次実行・変数定義）。再代入で型が
  変わる Ruby と、『しくみ』のシャドーイング省略を対比。

## Part 4. Union と制御フロー・ナローイング（『しくみ』に無い独自領域）

**狙い**：gradual 型推論器の心臓部の一つ。『しくみ』が踏み込まない領域だと明示する。

- `if`/`case`/`&&`/`||` で枝ごとに型が分かれる → `Union`。
- truthy/falsey エッジ、`nil?` / `is_a?` によるナローイング。FactStore のミニ版
  （フロー感応な事実集合）。
- ナローイングは*型代入ではなく事実の追加*として実装（Rigor 流）。
- サイドバー：**『しくみ』には対応章が無い**ことを述べ、『しくみ』 第5章演習「タグ付き union ＋
  narrowing」を*最も近い親戚*として紹介。ここから先が gradual の独自地形。

---

## Part 5. 構造的な型 ― HashShape と Tuple

**狙い**：ハッシュ／配列リテラルを構造的に型化。『しくみ』 第5章（オブジェクト型）の Ruby 版。

- ハッシュリテラル → `HashShape`（キーと値型）、配列 → `Tuple`。
- 要素アクセス（`h[:k]`, `a[0]`）の推論。
- **open vs closed**：Rigor は誤検知忌避で open 寄り。『しくみ』の「プロパティ完全一致 → 余剰を
  拒否」を*逆向きの設計判断*として対比し、「なぜ Rigor は許すのか」を議論。
- サイドバー：**『しくみ』 第5章**（構造的型・record）。

---

## Part 6. 部分型・整合性・三値判定

**狙い**：受理判定を二値から三値へ。『しくみ』 第7章（部分型付け・変性）＋ gradual。

- `accepts(other) -> yes | no | maybe`（Rigor の AcceptsResult / Trinary のミニ版）。
- 部分型 `<:`（値集合包含）と **gradual consistency**（`Dynamic` が絡むと対称・非推移）の
  両輪。
- 変性：返り共変・引数反変を軽く。join/meet は「Rigor は正規化（union）で実装する」と補足。
- サイドバー：**『しくみ』 第7章**（`typeEq → subtype` 差し替え、共変/反変が実装で向きを変える、
  「健全だが良いプログラムを弾くのは誤検知」のコラム＝Rigor 哲学の源流）。

## Part 7. RBS をひとさじ

**狙い**：手書きメソッドテーブルを、最小の RBS 読み込みに置き換える。Rigor 固有。

- コア型の RBS をごく一部読み込み、ディスパッチの戻り値・引数型に使う。
- 型変数の置換（`Array[Elem]` → `Array[String]`）の初歩。『しくみ』 第9章の `subst` の正道版。
- サイドバー：**『しくみ』 第9章**（型代入・α同値）。「Rigor は RBS 由来なので変数捕獲の露出面は
  小さい」と補足。

---

## Part 8. 型推論の心臓 ― 『しくみ』が避けた領域

**狙い**：chibirigor の主題。『しくみ』が「正解を知らない」と投げた推論に踏み込む。

- 戻り値推論（注釈なしメソッドの戻り型を本体から導出）。
- 引数からの簡易推論／双方向型付け（bidirectional）の初歩。
- 制約ベース推論や Hindley–Milner は*紹介に留め*、chibirigor では「自明なケースに限定」する
  方針（『しくみ』のおわりにの助言と同じ立場）。
- サイドバー：**『しくみ』 第9章演習＋おわりに**（推論は未解決の前線）。外部文献（双方向型付け、
  HM）をここで補う。

---

## Part 9. `annotate` を仕上げる

**狙い**：推論結果を人が読める形に。

- 推論シグネチャを RBS 風に、またはインラインコメントで出力。
- `check` と `annotate` が同じ推論器を共有することを示す（推論が主、検査は副産物）。

## Part 10. gradual の哲学を完成させる

**狙い**：全体を Rigor の価値観で締める。『しくみ』「おわりに」の gradual 伏線と接続。

- `Dynamic` の伝播規則（`Dynamic[A] | B = Dynamic[A|B]` 等）を仕上げ。
- 誤検知忌避の総括：「動くコードを脅かさない」を診断方針として明文化。baseline 的発想を紹介。
- 『しくみ』のおわりに（[ST06]/[SV07] への伏線）を引用し、「『しくみ』の終着点が chibirigor の出発点」で
  締める。

### Part 8. annotate を仕上げる（固有）
- 8-1 推論シグネチャを RBS 風 or インラインコメントで出力
- 8-2 `check` と `annotate` が同じ `type_of`/`accepts` を共有することを示す
- 8-3 `untyped` がどこに出るか＝推論の弱点の可視化（`sig-gen` の芽）
- 8-4 まとめ

### Part 9. まとめ：gradual の哲学（『しくみ』 おわりに）
- 9-1 `Dynamic` の伝播を仕上げ（`A | Dynamic` → `Dynamic` など）
- 9-2 「わざと見逃す 4 箇所」（`Dynamic`／open shape／`:maybe` 非昇格／保守的 narrowing）
- 9-3 baseline 的発想（既存の指摘を呑み、新規だけ見る）
- 9-4 『しくみ』おわりに（gradual への伏線）と接続：「『しくみ』の終着点が、chibirigor の出発点」

## 深掘り考察（v2）― 「推論器」の正体と、設計を貫く一本の軸

未決事項を個別に潰す前に、それらを*同時に*解く軸が無いかを考えた。結論：**双方向型付け
（bidirectional typing）を背骨に据えると、「推論器とは何か」「annotate は容易か」
「診断はいつ出るのか」「なぜ動くコードを脅かさないのか」がすべて一つの仕組みから導ける。**

### 考察1：Rigor は HM 推論器ではない ― 「推論器」を精密化する

`Part 0.1` で「chibirigor は型*推論器*」と書いたが、これは精密化が要る。世間で「型推論」と
聞いて想像される Hindley–Milner（HM, 大域的な制約を集めて単一化で解く）は、**Rigor の仕組み
ではない**。Rigor が「推論」と呼んでいるものは、実は次の 4 つの合成にすぎない：

1. **上向き合成（upward synthesis）**：リテラル／定数畳み込み。`1` → `Constant[1]`,
   `[1, "a"]` → `Tuple[Constant[1], Constant["a"]]`。『しくみ』の `typecheck` の*リッチ版*。
2. **フロー・ナローイング**：`if x.nil?` の枝で `x` の型を絞る（FactStore）。
3. **シェイプ推論**：ハッシュ／配列リテラルから `HashShape`/`Tuple` を起こす。
4. **シグネチャ参照（signature lookup）**：メソッドの戻り型・引数型は *RBS から引く*。
   ここは「推論」ではなく「参照」。

つまり Rigor の "推論" は**構文主導の上向き合成 ＋ フロー ＋ 参照**であって、メソッドの
引数・戻り型をコードから solve する HM とは別物。**この事実が `Part 0.1` の「検査器 vs
推論器」を一段精密化する**：chibirigor は「注釈を書かなくても局所的に型が付く（合成する）」
意味で推論器だが、「大域制約を解く」意味の推論器ではない。『しくみ』の検査器との本当の差は
「HM をやるかどうか」ではなく、**「注釈の無い箇所を合成で埋めるか、注釈必須にするか」**。

→ `Part 0.1` にこの精密化を反映する（HM への誤解を最初に解いておく）。

### 考察2：双方向型付けこそが正しい背骨（Part 8 の到達点が確定する）

双方向型付けは型付けを 2 方向に分ける：

- **合成 `⇒`（synthesize / infer）**：式から型を*上向きに*得る。『しくみ』の `typecheck`、Rigor の
  ExpressionTyper はこれ。注釈不要。
- **照合 `⇐`（check）**：*期待型*に対して式を*下向きに*検査する。呼び出し地点で実引数を
  RBS 引数型に対して `accepts` するのがこれ。

これが効く理由は二つ：

- **『しくみ』から地続き**。『しくみ』の `typecheck` はすでに純粋な `⇒`。chibirigor は「呼び出し地点と
  注釈箇所だけ `⇐` を足す」差分で済む。HM のような別パラダイムを持ち込まない。
- **robustness principle（Postel の法則）と完全一致**。返り型は `⇒` で*厳密に合成*（最も狭く）、
  引数は `⇐` で*寛容に照合*。Rigor の「返り厳密・引数寛容」は、双方向の二方向そのもの。

**→ 決定：`Part 8` の到達点は「双方向型付けの初歩」で確定。** 制約ベース／HM／単一化は
*紹介に留める*（『しくみ』おわりにの助言「自明なケースに限定せよ」と同じ立場）。これで「双方向で
止めるか HM まで行くか」の未決事項が解ける。

### 考察3：診断は `⇐` でしか出ない ― これが「脅かさない」の正体（headline）

双方向にすると、**型エラー（診断）は照合 `⇐` の位置でしか発生しない**ことが構造的に従う。
合成 `⇒` は「型を一つ作る」だけで、失敗しても診断にせず `Dynamic` を返せばよい（fail-soft）。

そして `⇐` の位置＝**期待型が宣言されている所**＝RBS シグネチャのある呼び出し・明示注釈・
宣言付き戻り値、に限られる。逆に言うと：

> **注釈の無いコードには期待型が無い → `⇐` 位置が存在しない → 診断が出ようがない。
> ゆえに「注釈の無い動くコードは、構造的に、決して脅かされない」。**

これが Rigor の "never frighten working code" の*形式的な裏付け*になる。チュートリアルで
最も価値ある一行はここ。`check` は「推論の副産物」だと `Part 0.1` で書いたが、より正確には
**「`check` ＝ 合成した型を `⇐` 位置で期待型に突き合わせる行為」**。`Part 6`（accepts/三値）と
`Part 10`（gradual 哲学）はこの一文に収束させる。

→ `Part 0` のどこかに「診断は `⇐` でしか出ない」を*明示的な原理*として置く。

### 考察4：`annotate` は容易か ―「容易なら」への回答

双方向で見ると `annotate` の難易度が綺麗に分解できる：

- **戻り型・ローカル型の annotate ＝ 合成 `⇒` の副産物**。すでに計算済みの型を出すだけ。**容易**。
- **引数型の annotate ＝ 本体での使われ方からの*逆方向*推論**が要る（Rigor の capability role の
  芽）。これは `⇒`/`⇐` のどちらでもなく usage-based で、難しい。

**→ 結論：「容易な範囲で yes」。** `annotate` の MVP は **戻り型＋ローカル型**を出す。注釈の
無い引数は `Dynamic`（または「`?`」）として出し、*引数型の推論は明示的に stretch goal*。
これは「容易であれば annotate も」というご要望にちょうど沿う線引き。出力形式は、まず人が
読む**インラインコメント or RBS 風**を MVP、機械可読 JSON は後回し（未決のまま据え置き）。

### 考察5：ブートストラップ問題 ― Ruby に「リテラルだけの島」は無い

『しくみ』は `1 + 2` を構文（add ノード）として直接型付けできるが、Ruby の `1 + 2` は
`Integer#+` の*メソッド送信*。つまり **chibirigor は Part 1 の時点で既に最小のメソッド
ディスパッチが要る**（リテラルだけで完結する島が無い）。選択肢：

- (A) Part 1–2 は**手書きの極小メソッドテーブル**（`Integer#+ -> Integer` 等を数件）→
  Part 7 で RBS に*置き換える*。
- (B) Part 2 で薄く RBS を読み、テーブルを最初から持たない。

**→ 決定：(A)。** 序盤に RBS パーサの複雑さを持ち込まない方が学習曲線が緩い。chibivue も
「まず手書き → 後で本物に差し替え」を多用する。`Part 7` の役割を「手書き表を RBS に
*differ-して* 置換し、置換しても診断が変わらないことを確かめる」と再定義する（=リファクタ
体験として設計）。

### 考察6：最小型格子の確定カットと「再帰型を作らない」決断

MVP の carrier を確定する：

- **MVP コア**：`Top` / `Bot` / `Dynamic` / `Constant` / `Nominal` / `Union`（＋ `Trinary`）。
- **Part 5 で追加**：`HashShape` / `Tuple`。
- **OUT（「実 Rigor にはある」と一言だけ）**：`Refined` / `Intersection` / `Difference` /
  `Singleton` / `DataClass`・`DataInstance` / `App`（HKT）/ `BoundMethod`。

帰結として **chibirigor では再帰型（『しくみ』 第8章）を*実装しない*。** 再帰型は Rigor では HKT/
`App`＋fuel で扱うが、それは OUT。よって『しくみ』 8 章は*理論サイドバー（読み物）*として紹介する
だけにする。これは正しい間引き：再帰型を入れると余帰納（coinduction）か fuel 予算が要り、
最小実装の主旨（双方向 ＋ gradual ＋ フロー）から逸れる。

### 考察7：「健全性を捨てる線」を見せる step を置く

chibirigor は意図的に unsound（健全でない）。*どこで*健全性を捨てるかを明示する step を置くと、
『しくみ』（健全・静的）との対比が最大化し、教育価値が跳ねる。捨てる箇所は 4 つ：

1. `Dynamic` は何でも受理する（境界の穴）。
2. open な `HashShape` は余剰キーを許す（『しくみ』は完全一致で拒否）。
3. `maybe`（三値）を `no` に*昇格させない*（疑わしきは罰せず）。
4. ナローイングは保守的に*緩める*側に倒す（再代入・クロージャ捕獲で事実を捨てる）。

→ `Part 10` に「健全性をどこで・なぜ捨てたか」の総括 step を追加する。

## 深掘り考察（v3）― 教材の狙い・トーン・複雑さの予算

サブテーマが固まった：chibirigor は**「型理論 ↔ Ruby/RBS ↔ Rigor 実装の問題」を自然に結ぶ
やさしい導入**。この一点が、v2 のやや上級寄りな書きぶりを*プレゼンと実装の両面で軽量化*する
基準になる。

### 考察8：編集原理 ―「三題噺」を各 step の背骨にする

各 step を次の小さな弧で書く。これがサブテーマを*構造*に落とす方法：

1. **型理論（『しくみ』から）**：1 つの概念だけ。やさしく、できれば 1 例で。
2. **Ruby/RBS ではどう見えるか**：その概念が Ruby だとどう現れる／*現れない*か、RBS は何を
   言える／言えないか。
3. **Rigor が実装で困った所**：その素直な実装が現実の Ruby でなぜ破綻するか、Rigor がどう
   折り合ったか。← ここが「導入として Rigor を理解させる」核心。

**各 Part の三題噺（早見表）**：

| Part | ① 型理論（『しくみ』） | ② Ruby/RBS での見え方 | ③ Rigor 実装の問題 |
|---|---|---|---|
| 1 リテラル/算術 | 値に型を与える・型の内部表現（2章） | `1` は `Integer`。RBS はそこ止まり | リテラル型 `Constant[1]` が*増えすぎる*→いつ `Integer` に丸めるか（表示予算・正規化） |
| 2 メソッド送信 | 関数の型（3章） | Ruby は `+` すら*メソッド送信*。型は RBS が与える | オープンクラス／継承チェーン解決、未知メソッドは脅かさず `Dynamic` に倒す |
| 3 Scope/文 | 型環境 tyenv（3・4章） | 再代入で型が変わる、ブロックが外側を捕獲 | 不変 Scope、捕獲時の事実無効化（緩める側に倒す） |
| 4 Union/ナローイング | （『しくみ』に無い）場合分け | 偽は `false`/`nil` だけ、`is_a?`/`nil?`/`case` | 絞り込みで誤検知を出さない、`case` の到達不能枝（ADR-47） |
| 5 HashShape/Tuple | レコード型（5章） | ハッシュは動的、symbol キーの options が氾濫、RBS の record | open か closed か（完全一致で弾くと誤検知の嵐） |
| 6 受理判定/三値 | 部分型付け・変性（7章） | ダックタイピング、`untyped` が混ざる gradual | `yes/no/maybe` の三値、`maybe` を罰しない、robustness（返り厳密/引数寛容） |
| 7 RBS ひとさじ | 型代入（9章） | 型は*別ファイル*（.rbs）、コードに注釈が無いのが普通 | RBS スーパーセット、`erasure` で RBS に戻せること |
| 8 推論 | 型推論（『しくみ』が避けた前線） | 引数に型が無い、どう型を付ける？ | 引数は `Dynamic` に倒す or capability で粗く推論＝*未解決の前線* |
| 9 annotate | 合成した型を見せる | sig-gen 的発想 | 推論の弱い所が可視化される（ギャップ自体が価値） |
| 10 gradual 哲学 | 健全性 vs gradual（おわりに） | 動的言語への後付けの型 | 「動くコードを脅かさない」を設計価値に、baseline |

この表はそのまま各 step の見出し設計に使える。**「Rigor を理解してもらう」とは、③ の問題が
②（Ruby の現実）から*必然的に*生まれることを、① の概念で*やさしく*腑に落とすこと**だと
定義しておく。

### 考察9：トーン ―「易しさ」を担保する具体ルール

- **数式・ギリシャ文字を本文に出さない**。`⇒`/`⇐` や μ・α同値・共変/反変などは、まず
  *普通の言葉*で説明し、用語は「実はこれ◯◯と呼ばれます」とコラムで後出し。
- **双方向型付けは“用語より先に仕組み”**。`type_of`（型を求める）と `accepts`（期待型に
  合うか確かめる）という*2 つの関数*を素朴に作るだけにして、「この 2 方向、まとめて
  *双方向型付け*と言います」は Part 6 か巻末コラムで明かす。← v2 の核心を保ったまま易しくする。
- **1 step ＝ 概念 1 つ＋実コード 1 まとまり**。『しくみ』が `typecheck` に case を 1 つ足す粒度に
  倣う。
- **「動くコードで確かめる」を毎 step**。『しくみ』のように最後に必ず実 Ruby を流して挙動を見せる。

### 考察10：実装の複雑さ予算 ―「複雑にしない」を仕様で縛る

「複雑になりすぎない」を願望でなく*制約*にするための線引き：

- **1 step に難所は 1 つ**。新カリア追加 *か* 新ルール *か* 新ディスパッチ源、のどれか 1 つ。
  2 つの難しいことを同時に入れない。
- **データ構造は素朴に**：`Scope` は注釈付き Hash 数本、`FactStore` は*ただの事実リスト*
  （実 Rigor の 6 バケツ・stability は持たない）、`accepts` は `:yes/:no/:maybe` を返すだけ。
- **不実装を最初から宣言**（実 Rigor にはあると一言だけ）：HKT/`App`・fuel、**再帰型**
  （→ 『しくみ』 8 章は読み物サイドバー）、`Refined`/`Intersection`/`Difference`/`Singleton`/
  `DataClass`、プラグイン、キャッシュ、LSP、インクリメンタル、性能最適化、健全性証明。
- **RBS は“ひとさじ”**：フル RBS パーサは作らない。コア型の戻り値・引数を引ける最小限だけ。
- **推論（Part 8）は“自明な範囲”で止める**：引数は基本 `Dynamic`。usage 推論（capability）は
  *おまけ step* に隔離し、本線を重くしない。

この予算により、最小実装は「Prism ＋ 数百行の型推論器」に収まる見込み（『しくみ』の `typecheck`
≈ 40 行が、gradual ＋ フロー ＋ ディスパッチ分だけ膨らんだ規模感）。

### 考察11：v2 との折り合い（上級ネタの“格下げ”）

v2 で出した上級トピックは*捨てず、置き場所を下げる*：

- 双方向 `⇒`/`⇐`：仕組みは本線、*用語と記法*はコラム（考察9）。
- robustness principle：Part 6 で「返りは厳しめ・引数は緩め、なぜなら脅かさないため」と
  *日常語*で。法則名は脚注。
- 「診断は `⇐` でしか出ない＝注釈の無いコードは脅かされない」：これは**やさしいまま強い**ので
  本線の山場として残す（考察3）。専門用語ゼロで言える。
- 健全性を捨てた箇所（考察7）：Part 10 で「Rigor はわざと*見逃す*ことで動くコードをいじめない」
  と物語的に。形式語は出さない。

### 考察12：本編／続編の二分割（Little/Seasoned 方式）

『Scheme 手習い（The Little Schemer）』→『Scheme 修行（The Seasoned Schemer）』のように、
**最初に構想した高度な内容は続編に送る**。ただし鉄則を一つ置く：

> **「Rigor の基礎を理解するのに*必要*な内容」は、たとえ少し難しく見えても本編に残す。**
> 続編に送ってよいのは「*作り込み*や*本格的な型理論*」であって、「*なぜそう設計されているか*の
> 腑落ちに要る話」ではない。

（※ Little/Seasoned の Q&A 文体は**参考にしない**。普通の解説文で書く。）

**本編「The Little chibirigor」＝やさしい導入（check ＋ annotate が動くまで）**

| 本編 Part | 内容 | 残す理由（基礎理解に必須か） |
|---|---|---|
| 0 イントロ | 検査器 vs 推論器、拒まない入力、`type_of`/`accepts` の 2 関数、「脅かさない」原理 | Rigor の世界観そのもの。専門用語ゼロで言える |
| 1 リテラル/算術 | `Const`・`type_of`・丸め問題 | 型を作る最小の一巡 |
| 2 メソッド送信 | 手書きディスパッチ表、未知は `Dynamic` | Ruby＝全部メソッド送信、の核 |
| 3 Scope/文 | 不変 Scope、再代入 | 型環境の基礎 |
| 4 Union/ナローイング | 場合分けで型が割れる、誤検知回避 | gradual の独自地形（やさしい範囲） |
| 5 HashShape/Tuple | open か closed か | 構造的型の基礎＋誤検知の実感 |
| 6 受理判定・三値・gradual | `accepts`＝`:yes/:no/:maybe`、`Dynamic`、robustness を*日常語*で、「脅かさない」山場 | **gradual の心臓。基礎理解に必須なので本編に残す** |
| 7 RBS ひとさじ | 手書き表を最小 RBS に置換 | 「型は別ファイル」という Ruby/RBS の基礎事実 |
| 8 annotate を仕上げる | 合成型を見せる、推論の弱点が可視化される | 推論結果を*見る*体験。容易 |
| 9 まとめ：gradual の哲学 | わざと見逃す、baseline、『しくみ』おわりにと接続 | 締め。物語で |

**続編「The Seasoned chibirigor」＝高度な型理論・作り込み**

- 双方向型付けの*形式化*（`⇒`/`⇒` の記法・メタ理論）。本編では用語を出さず仕組みだけ。
- 部分型付けと**変性**の形式（関数引数の反変など）。本編は「返り厳しめ・引数緩め」の直感止まり。
- **再帰型**（μ・余帰納）↔ Rigor の HKT/`App`＋fuel。本編は非実装（『しくみ』 8 章は読み物）。
- **本物の型推論**（制約ベース、capability/duck 推論で*引数*を埋める）。本編は引数＝`Dynamic`。
- **完全な FactStore**（6 バケツ・stability・クロージャ捕獲の機微）。本編は素朴な事実リスト。
- 健全性／正規化の*理論*。本編は「わざと見逃す」物語で。
- 型代入・ジェネリクス具体化・`erasure` の形式。本編は RBS「ひとさじ」。
- 実 Rigor の作り込み（プラグイン・キャッシュ・LSP・性能）。

**境界線の覚え方**：本編＝「Rigor を*使える*ように・*なぜそう作るか*が腑に落ちるまで」。
続編＝「Rigor を*作れる*ように・型理論を*本格的に*」。考察3 の「注釈の無いコードは脅かされ
ない」やrobustness は*基礎理解側*なので、難しそうでも本編に残す（やさしく語れるから）。

### 考察13：`annotate` の出力形式（決定）

候補は 3 つ。教材としての価値で選ぶ：

| 候補 | 長所 | 短所 | 判定 |
|---|---|---|---|
| **インライン注記**（行末 `# => 型`） | 『しくみ』の `console.log` と同じ「見て確かめる」体験、行番号があるので*実装は数行*、型が値の隣に現れる満足感 | 機械処理向きではない | **MVP に採用** |
| **RBS 風**（メソッドの sig を出力） | Ruby/RBS の生態系と接続（Part 7 の RBS、`erasure` の伏線）、「`untyped` がギャップを示す」が映える | メソッド単位＝*引数*の推論が要り、本編では引数＝`Dynamic` | **Part 8 の二次出力**に採用（戻り型は合成済み、引数は `untyped` 表示でギャップを*見せる*） |
| **JSON** | LSP・ツール連携向き | 教材では読みにくい | **続編**（LSP の節）へ |

**決定**：

- **MVP（Part 1）＝インライン注記**。`type_of` の結果を行末に `# => 型` で出すだけ。『しくみ』の
  「最後に必ず実 Ruby を流して結果を見る」に揃う、一番やさしい形。
- **Part 8 ＝ RBS 風のメソッド sig 出力を追加**。戻り型は合成済みなのでそのまま、引数は
  わからなければ `untyped` と出す。**この `untyped` の出方そのものが「Rigor が型を見失った場所＝
  推論の弱点」の可視化**で、`sig-gen` の発想の芽になる（考察4・三題噺 Part 8 の③）。
- **JSON は続編**（LSP）。

これで「`annotate` の出力形式」未決は解消。引数型の usage 推論（capability）は引き続き*続編*。
