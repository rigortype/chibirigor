# Java 読者レビュー（説明の飛躍を炙り出す）

## ペルソナ宣言

私は **Java で普通に機能開発ができる中堅エンジニア**。本書の最大ボリュームの読者層
＝「静的型付け言語の実務経験はあるが、型理論は未学習」の代理人としてこのレビューを書く。

私の知識の境界（厳守）：

- ジェネリクス（`List<String>`）は**なんとなく使える**が、**「型消去（type erasure）」という語を知らない**。
  ジェネリクスが実行時にどう実装されているかは考えたことがない。
- `NullPointerException`（ぬるぽ）は日常的に踏むが、**それを型システムの言葉で説明できない**
  （null 安全・Option/Maybe 型・健全性の欠陥…という接続は頭に無い）。
- 部分型・変性・健全性・gradual typing・双方向型検査・単一化・HM・System F・型消去・
  代数的データ型…といった型理論の専門語は**すべて初見**。
- 一方で **Java の足場**は持つ：継承、インターフェース、`instanceof`、キャスト、
  オーバーロード、`Object`、null、例外、`extends`/`super` のジェネリクス境界（うっすら）。
- Ruby の文法は読めば追える程度。

専門語が定義なしに出たら「ここで詰まった」と正直に申告するのが私の仕事。**私の無知は資産**。

## 総評

**結論：前編は「型理論ゼロの Java 読者が、ほぼ飛躍なく読める」。完成度は高い。** ただし
**1 つの致命的な語（erasure）** と、**2 つの惜しい未架橋（NPE、部分型 vs 継承）** が残る。

良かった点を先に言う。本書は Java 読者の直感ズレを**意識的に橋渡ししている箇所が多い**。
特に Part 1 の `any`/`unknown` コラム、Part 4 の「missing arm vs unreachable arm（Java/C#
の網羅性との方向の違い）」コラム、Part 4 の `never`/ボトム型の言語対応表、Part 6 の Postel
コラムは、私のような読者を明確に名指しして書かれていて、読んでいて「自分のために書かれている」
と感じた。専門語を出すときに「名前は覚えなくて大丈夫」「用語は後で」と添える作法も、
詰まりを未然に防いでいる。三題噺の②（Ruby だと）が毎章あるおかげで、抽象が地に足を付く。

致命傷は **`erasure`（型消去）という語が前編に 4 回出てくるのに、一度も定義されない**こと
（Part1/7/8/9）。私はこの語を知らない。しかも**2 つの違う意味で使われている**ように読めて、
混乱した（後述 STUCK-1）。これは「型理論ゼロで読める」という前編の価値を直接傷つける。

惜しい未架橋が 2 つ。(1) Part 4 の `User | nil` ナローイングは、**まさに私が毎日踏む NPE
（ぬるぽ）の話**なのに、本文は一度も NPE/null 安全に触れない。「Java で null を `if != null`
で守るあれが、型の世界ではこう表現される」と一言あれば、私の実務経験と直結して一気に
腑に落ちたはず（CONFUSED-1）。(2) **「部分型（subtype）」が継承との異同を説明せずに導入される**
（Part 6）。Java 育ちの私は「部分型＝継承（extends）」と素朴に同一視してしまい、しかも
Part 5 の HashShape（構造で部分型が決まる、継承と無関係）で自分の思い込みが静かに破綻する。
本文はそこを名指しで埋めてくれない（CONFUSED-2）。

これら 3 つを直せば、前編は「型理論ゼロの Java 読者が飛躍なく読める」を達成すると判断する。

## 特に効く修正トップ5

1. **【最優先・STUCK】`erasure`（型消去）を前編から一掃 or 一度だけ平易に定義する。**
   現状 4 回（Part1/7/8/9）無定義で登場し、しかも 2 つの意味（①境界で精度を捨てる、②型を
   RBS に書き出す）に割れて見える。Java 読者は「型消去」を知らず、ジェネリクスの実装の話とも
   区別できない。最善は「前編では erasure という語を使わず『境界で保守的な型に落とす』『RBS に
   書き出す（sig-gen）』と日本語で言い分け、erasure の正式名は後編 Part 6 に集約」。残すなら
   用語集に項を立て、初出（Part 1）で「型情報を落として保守的な型にすること。Java のジェネリクス
   型消去とは別物」と一文で定義する。

2. **【CONFUSED→実務直結】Part 4 のナローイングを NullPointerException（ぬるぽ）に明示接続する。**
   `User | nil` の話は Java 読者が毎日踏む NPE そのもの。「Ruby の `nil` ＝ Java の `null`」
   「`if (x != null)` で守るあの習慣を、型が追えるようにしたのがナローイング」「`nil` を絞り切れない
   場所こそ、Java なら NPE が出る場所」と一段落足すだけで、未学習の読者が「null は型で防げる
   種類のバグだ」と初めて気づける。本書の山場の価値が跳ね上がる。

3. **【CONFUSED→直感のズレ】「部分型 ≠ 継承だけ」を Part 5 か Part 6 で明示する。**
   Java 育ちは「部分型＝継承（子は親の部分型）」と思い込む。ところが Part 5 の HashShape は
   構造（キーの多寡）で部分型が決まり、継承と無関係。「キーが多い方が部分型」は直感に反する。
   「Java の継承による部分型に加えて、構造で決まる部分型がある。`Integer` を `Object` に入れられる
   あれも部分型」と、知っている継承と明示的に結ぶ一文を、部分型を重く使い始める直前に置く。

4. **【nitpick→配置】Part 1 の `any`/`unknown` コラムの密度を下げるか、後ろに送る。**
   情報は良質だが「格子」「部分型」「Top/Bot」を、本編でそれらを定義する前に投下している。
   前編 Part 1 の段階では `unknown`/`any` の「絞れと迫る vs 黙る」の対比だけに絞り、格子・
   双対の話は Part 4（ボトム型コラム）以降か後編に寄せると、初読の気後れが減る。

5. **【nitpick→足場提示】言語対応表に Java の欄/不在を明記する。**
   Part 4 の `never`/ボトム型表、Part 1 の `untyped` 表などで、Java 読者は「自分の言語はどこ？」を
   探す。`never` は Java に相当物が無いなど「Java には無い（だから新概念）」と一言添えると、
   読者は自分の足場との距離を測れて安心する（Part 8 の `void` 区別は既に良い手本）。
